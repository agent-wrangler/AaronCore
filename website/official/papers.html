Working papers

Theses from building continuity into an agent

These are not neutral docs. They are position papers from the AaronCore direction: memory as runtime, continuity as a systems problem, and state as the thing that keeps work from dissolving between turns.

What this page is for

Product pages explain what AaronCore does. This page explains what it believes. The target is not a feature checklist. The target is a clearer vocabulary for what current agents still keep breaking.

Paper 01 Memory architecture

Memory is not a feature

Most systems still talk about memory as if it were a plugin. The assistant answers first, then reaches backward for a note, a retrieval hit, or a thin user profile. The result is familiar: the product can claim memory, but the conversation still feels restarted every time it matters.

AaronCore takes the opposite posture. Memory is not something added after thought. It is part of the runtime that shapes what counts as the current situation. That changes the design question. The goal stops being "how much can we store?" and becomes "what memory should participate in this turn, and through which layer?"

Once memory is treated as runtime, layers matter. Recent dialogue, session understanding, long-horizon experience, persona rules, execution traces, and learned methods should not collapse into one bucket. They carry different authority. They decay differently. They should come back through different paths.

This is why raw retrieval is not enough. Search can fetch text. It cannot decide whether the current moment calls for a flashback, a recap, a user-preference reminder, or a task-level continuation. The point of memory is not recall for its own sake. The point is to alter what happens next.

Paper 02 Agent continuity

Continuity breaks before intelligence

A lot of frustration with assistants gets misdiagnosed as an intelligence problem. Users say the model is shallow, forgetful, unreliable, or not serious enough. Often the underlying failure is more basic: the system lost continuity before intelligence ever had a chance to matter.

An assistant can be locally impressive and still globally discontinuous. It can answer one turn well, summarize a PDF, write a script, or call a tool. But if the same system wakes up like a stranger on the next turn, the user's sense of progress collapses. The work has to be re-explained. The relationship has to be re-established. The thread thins out into fragments.

This is why continuity should be treated as a first-order systems concern, not a nice-to-have. It sits between memory and action. It decides whether the current reply belongs to a live trajectory or to yet another isolated completion. Without continuity, even a strong model feels like an endless series of amnesic resets.

AaronCore is trying to push on exactly this seam. Not by promising omniscience, but by refusing to accept restart as the default shape of interaction. The more ambitious claim is not "the model knows more." The more ambitious claim is "the work can keep going."

Paper 03 Runtime state

State beats prompt tricks

Prompting can shape tone, style, and immediate behavior. It is much worse at sustaining work across time. As soon as continuation depends on wording alone, the system starts leaning on heuristics: detect a keyword, assume a follow-up, guess the user's intention from the last line. This looks clever until the moment the thread branches, stalls, or resumes after interruption.

Long-running work needs anchors stronger than phrasing. It needs explicit state: what task is active, what step was last attempted, what blocked, what changed, what still needs verification, and what target the system believes it is acting on. Without that substrate, "continue" becomes theater.

This is one of AaronCore's strongest design commitments. Let the model reason about the next move, but let runtime state hold the continuity. Plans, stores, execution traces, blockers, and verification should live as system facts, not as fragile prompt residue.

The payoff is not only reliability. It is also honesty. Once state is explicit, the system can say what it knows, what it tried, what failed, and why the next step follows. That makes continuation legible. And legibility matters, because trust does not come from elegant prose. It comes from a path the user can actually inspect.
