# 聊天消息间距与段落渲染经验

最后更新：2026-04-02

这份记录只说明一件事：为什么聊天消息曾经“越调越松”，以及后续应该改哪里、不该改哪里。

## 现象

- 用户感知是“每一句和每一句之间太空”。
- 表面看像 CSS 问题，但实际往往是三层叠加：
  1. LLM 回复里本来就有空行。
  2. `static/js/utils.js` 旧逻辑把每一行普通文本都渲染成独立 `.bubble-p`。
  3. `static/js/chat.js` 流式阶段还会按句号把文本提前拆成多个 reply part。
- 还有一个常见假象：旧消息可能已经被缓存成现成 HTML 快照，新的解析逻辑不会自动重跑。
- 还有一个常见假象：历史重渲看起来正常，但实时流式消息如果没有在结束时合并回整条消息，也会和历史排版不一致。

只改 `static/css/main.css` 往往只能缓解，不能根治。

## 这次验证过的正确边界

### 1. 最终渲染层：先看 `static/js/utils.js`

关键点：

- 普通文本应按“真实段落”合并，而不是一行一个块。
- 空白行应该只作为段落结束符，不应该额外生成 `.bubble-spacer`。
- 同一段里的多行文本可以用 `<br>` 保留轻微换行感。
- 列表项之间如果只是空一行，仍应保持在同一个 `ol/ul` 里，不要拆成多个单项列表。

当前修复点：

- `formatBubbleText(...)`
- `flushParagraph()`
- 空行分支：`if(!trimmed){ flushParagraph(); flushList(); continue; }`
- 列表判断：空行后如果还是同类型列表项，则继续当前列表

这层决定“最终一条消息渲染成几个段落”。

### 2. 流式渲染层：再看 `static/js/chat.js`

关键点：

- reply part 只应该在“显式换行”或“视觉换行溢出”时产生。
- 不要按句号、问号、分号把一句句硬切成独立小消息。
- 不要把前导空行升格成空白的 reply part。
- 流式结束后，必须把所有 reply part 合并回一条完整消息，再统一走 `formatBubbleText(...)`。

当前修复点：

- `_splitStreamFragments(...)`
- 去掉句号级别切段
- 跳过前导空行

这层决定“流式过程中为什么看起来像很多小段往上顶”。

### 3. 样式层：最后才调 `static/css/main.css`

只有前两层确认正常后，CSS 才值得微调。

当前比较稳的方向：

- `.msg.assistant .bubble-p`
- `.msg.assistant .bubble-p + .bubble-p`
- `.msg.assistant .bubble-ul, .msg.assistant .bubble-ol`
- `.msg.assistant .bubble-quote`

原则：

- 先修段落结构，再修视觉间距。
- 只做半档调整，避免一下子从“太松”走到“太挤”。

## 以后不要再这样改

- 不要看到“段落太松”就只改 `.bubble-p margin`。
- 不要重新把空白行渲染成 `.bubble-spacer`。
- 不要恢复“按句号自动拆 reply part”的逻辑。
- 不要同时大幅修改 `line-height` 和 `margin`，这样很难判断到底是哪层出了问题。

## 推荐排查顺序

1. 先看原始回复文本里是不是自带很多空行。
2. 再看 `formatBubbleText(...)` 是否把普通行拆成了多个 `.bubble-p`。
3. 再看 `_splitStreamFragments(...)` 是否把一句句提前拆成了多个流式块。
4. 再确认当前消息是不是旧的 `nova_chat_history` 快照，而不是新规则重渲出来的 DOM。
5. 最后再看 CSS 的段距和行高。

## 这次相关文件

- `static/js/utils.js`
- `static/js/chat.js`
- `static/css/main.css`
- `output.html`

`output.html` 的作用主要是前端资源版本号；当 JS/CSS 行为变了但界面没变，优先检查缓存版本。

另外，聊天历史快照现在带有渲染版本：

- `nova_chat_history`
- `nova_chat_history_render_version`

当 `formatBubbleText(...)` 或历史渲染结构发生变化时，应同步提升渲染版本，避免旧 HTML 快照绕过新逻辑。

## 新结论：标准 Markdown 才是最终渲染源

2026-04-02 这次又确认了一次：

- 前端 `formatBubbleText(...)` 更适合作为流式中的临时 fallback，不适合作为长期唯一 Markdown 解析器。
- 仓库后端已经有 CommonMark 渲染能力，应复用现有实现，而不是继续在前端手写列表和段落边界。
- 历史消息应由 `/history` 返回 `content_html`。
- 实时消息应由 `/chat` 的 `reply` 事件返回 `reply_html`。
- 前端最终态只负责展示标准 HTML，再由 `static/css/main.css` 给 `p`、`h1-h3`、`ul/ol`、`blockquote`、`pre/code` 做主题样式映射。

这样可以把“结构正确”和“视觉微调”真正分层：

1. CommonMark 负责段落、列表、标题、引用、代码块结构。
2. CSS 只负责层次、留白、主题色。
3. 流式阶段可以临时拆块，但结束后必须回到同一份标准 HTML。
