# 路由优化方案

> 最后更新：2026-03-16
> 给新对话的上下文文件。包含：当前路由架构、已完成的改动、遗留问题、下一步方案。

## 一、路由架构概览（当前）

路由决定用户消息走"LLM 聊天"还是"技能执行"。已从"关键词触发"升级为"意图分层 + 执行许可"。

```
用户输入
  │
  ├─ 阶段 0：硬编码上下文规则（故事追问、新闻缺失检测）
  │     → 命中则直接返回，不走后续
  │
  ├─ 阶段 1：关键词路由 core/router.py → route()
  │     ├─ 特殊检测层（笑话、股票、纠偏、文章选择等 _looks_like_xxx）
  │     │     → 命中则直接返回，不受意图分类影响
  │     ├─ _score_text() 计算 chat_score / skill_score（含反例库过滤）
  │     ├─ ★ classify_intent() 意图门控 ← 新增
  │     │     → discuss/inform/chat → 直接走 chat，跳过技能匹配
  │     │     → task → 允许进入技能匹配
  │     └─ 技能候选层：根据关键词具体程度分级置信度
  │           → confidence >= 0.9 直接走，不问 LLM
  │
  ├─ 阶段 2：轻量 LLM 裁决 route_resolver.py → llm_route_lite()
  │     → ~200 token prompt，带 L2 场景摘要 + 可用技能列表
  │     → 如果阶段 1 有候选（confidence >= 0.7），作为 hint 传给 LLM
  │     → ★ 意图门控：discuss/inform 且 conf >= 0.75 时直接返回，不让 LLM 翻盘
  │     → LLM 返回 {"intent": "chat 或技能名", "reason": "..."}
  │
  └─ 阶段 3：兜底（★ 保守回退）
        → lite 失败时：只有 confidence >= 0.85 才信关键词结果
        → 否则默认走 chat（不确定时宁可不执行）
        → 都没有则走重量级 LLM 路由（完整 L1-L8 上下文）
```

涉及文件：
- `core/router.py` — 关键词评分 + 意图分类 + 特殊检测（阶段 1）
- `core/route_resolver.py` — 多阶段编排 + LLM 路由 + 意图门控（阶段 0/2/3）
- `core/skills/__init__.py` — 技能注册（含 anti_keywords 加载）
- `core/skills/*.json` — 各技能元数据（含 anti_keywords 反例库）
- `agent_final.py` — POST /chat 端点，调用 resolve_route()

## 二、踩过的坑（debug 记录）

1. **"常州天气怎么样"回退到 chat**：classify_intent 加了组合证据门槛后，这个输入只有 1 项证据（技能关键词），被判为 chat。修复：多个技能关键词命中（skill_score >= 2.0）直接算 evidence = 2。
2. **"如果要画海报的话"误触发 draw**：discuss 检测被绕过（同时有任务词"画"+技能关键词"海报"）。修复：新增 STRONG_HYPOTHETICAL（如果/假如/万一），无条件返回 discuss。
3. **LLM lite 翻盘 discuss 分类**："头脑风暴吧 或者玩游戏都可以" router 返回 chat/discuss（conf 0.85），但 0.85 < 0.9 进了 LLM lite，LLM 返回 story。修复：在 resolve_route 加意图门控，discuss/inform + conf >= 0.75 直接返回。
4. **anti_keywords 不生效**：`core/skills/__init__.py` 注册时没提取 anti_keywords 字段。修复：加 `'anti_keywords': meta.get('anti_keywords', [])`。
5. **改代码后线上不生效**：Python 模块导入在启动时绑定，改了 router.py / skill JSON 必须重启 agent_final.py 进程。

## 三、已完成的改动

### 第一轮：短期止血（2026-03-16 上午）

#### 2.1 TASK_WORDS 瘦身
原来 `TASK_WORDS` 包含 `做/写/讲/看/来个/想听/想看/想要` 这些日常口语词，导致随便说句话就加 skill_score。

改后拆成两组：
- **强任务词** TASK_WORDS：`查/生成/画/打开/搜索/搜/帮我/给我/再来一个/换一个/长一点/继续讲`
- **弱任务词** WEAK_TASK_WORDS：`做/写/讲/看/来个/想听/想看/想要` — 单独不加分，必须搭配技能关键词命中才 +0.5

#### 2.2 短关键词降权
2 字关键词（如"游戏""故事""天气"）从 +2.0 降到 +1.0。
3 字及以上关键词（如"贪吃蛇""写篇文章"）保持 +2.0。

效果：纯短关键词（如只说"游戏"）skill_score 只有 1.0，进不了 `>= 2.0` 的技能候选门槛。

#### 2.3 建议/讨论句式检测
新增 `_is_suggestion_or_discussion()` 函数，检测讨论句式。
命中后：chat_score +1.5，skill_score *= 0.5

#### 2.4 天气关键词收紧
weather.json 里城市名作为关键词，导致"我在常州"触发天气查询。
改后：城市名命中时，必须同时存在天气动作词（天气/气温/温度/下雨/晴天/多少度等）才算。

#### 2.5 LLM lite hint 过滤
route_resolver.py 里，只有 confidence >= 0.7 的关键词候选才传给 LLM lite 当 hint。

#### 2.6 弱技能信号分流
- 有 matched_skill → hybrid 0.45（低置信度，交 LLM 确认）
- 无 matched_skill（只有任务词没有技能关键词）→ chat 0.7

### 第二轮：意图分层 + 执行许可（2026-03-16 下午）

#### 2.7 新增 classify_intent() — core/router.py
在 `route()` 的通用技能候选层前加意图门控，把用户输入分为 4 类：

| 意图 | 含义 | 示例 | 后续处理 |
|------|------|------|----------|
| `task` | 明确执行请求 | "帮我查天气""写篇文章" | 允许进技能匹配 |
| `discuss` | 讨论/建议/假设 | "或者玩游戏都可以""如果要画海报" | 走 chat，不进技能 |
| `inform` | 陈述事实/提供信息 | "我在常州""我叫彬哥" | 走 chat，不进技能 |
| `chat` | 闲聊/问候/情感 | "你好""哈哈""累了" | 走 chat |

分类规则（纯规则，不调 LLM）：
1. 强假设词（如果/假如/万一）→ 无条件返回 discuss
2. 讨论词命中 且 不同时有(任务词+技能关键词) → discuss
3. 探询词命中 且 无任务词 且 无技能关键词 → discuss
4. 自述模式（我在X/我叫X）且无任务词 → inform
5. 组合证据 >= 2 项 → task（见下方 2.8）
6. 以上都不是 → chat

#### 2.8 组合证据门槛
`task` 意图需要同时满足至少 2 项：
- A. 有动作词（TASK_WORDS）
- B. 有请求结构（帮我/给我/请你/帮忙/一下/能不能帮/可以帮）
- C. 有明确技能关键词命中（matched_skill != None）

特殊加分：
- 弱任务词 + 技能关键词 → evidence 至少 2
- 多个技能关键词命中（skill_score >= 2.0）→ evidence 至少 2（解决"常州天气怎么样"）

#### 2.9 反例库（anti_keywords）
每个技能 JSON 新增 `anti_keywords` 字段，命中时整个技能跳过匹配。

已添加反例的技能：
- `story.json`：讲个道理、故事性、故事结构等
- `weather.json`：我在北京、天气功能、做个天气等
- `run_code.json`：游戏行业、游戏好不好玩、或者玩游戏等
- `draw.json`：海报设计、画图软件、如果要画等
- `news.json`：新闻行业、新闻学、做个新闻等
- `stock.json`：股票市场怎么样、炒股技巧、如果买股票等
- `article.json`：文章结构、文章怎么写、如果写文章等

代码改动：
- `core/skills/__init__.py`：注册时加载 `anti_keywords` 字段
- `core/router.py` `_score_text()`：遍历技能时先检查 anti_keywords，命中则 continue

#### 2.10 意图门控 — core/route_resolver.py
在 resolve_route() 阶段 1 之后、阶段 2 之前，新增意图门控：
- 如果 core_route 的 intent 是 discuss 或 inform，且 confidence >= 0.75
- 直接返回 core_route，不让 LLM lite 翻盘

解决问题："头脑风暴吧 或者玩游戏都可以" router 正确返回 chat/discuss（conf 0.85），但 0.85 < 0.9 进了 LLM lite，LLM 返回 story。加门控后直接走 chat。

#### 2.11 保守兜底 — core/route_resolver.py
阶段 3 兜底改为保守回退：
- LLM lite 失败时，只有 confidence >= 0.85 才信关键词结果
- 否则默认走 chat（不确定时宁可不执行，也不要误执行）

## 五、未完成的优化（后续方向）

### 5.1 句子成分拆解（来自 GPT 方案 3.2）
当前路由只做"有没有某个词"，不区分词在句子中的角色。

目标：把用户输入拆成 背景 / 参数 / 动作 三部分：
- "我在常州，帮我查下天气" → 背景="我在常州"，动作="查天气"，参数="常州"
- "如果要画海报的话" → 背景="如果要画海报"，动作=无

实现思路：用简单的标点/连词分句，对每个子句独立做意图判断。

### 5.2 执行与回复分离（来自 GPT 方案 3.5）
当前技能执行和回复生成是耦合的。

目标：技能只返回结构化数据，回复由人格层统一润色。
- 好处：技能结果可以被人格层包装，语气更自然
- 当前状态：部分技能（news）已做了分离，其他技能还是直接返回文本

### 5.3 LLM 路由质量提升
llm_route_lite 的 prompt 比较简单，LLM 有时判断不准。可以考虑：
- 给 LLM 更多 few-shot 示例
- 用更结构化的 prompt 格式
- 对 LLM 返回结果做二次校验
