# 自主学习、修复提案与审批说明
最后更新：2026-03-14

## 这份文档回答什么

这份文档只回答设置页里和“学习、修复、审批”有关的 4 类问题：

1. 设置页现在到底有哪些入口。
2. 每个卡片、按钮、开关分别控制什么。
3. 修复提案是怎么生成、怎么进入审批、怎么应用、怎么回滚的。
4. 为什么现在只保留一个显式审批入口。

## 当前设置页的用户可见区块

当前设置页对外主要是 5 个区块：

1. `自主学习`
2. `学习强度`
3. `知识转化`
4. `修复提案审核`
5. `开发者细调`

前 4 个是普通使用时应该直接看得到、看得懂的区块。
`开发者细调` 是底层策略区，不是第二套用户流程。

## 1. 自主学习

这一块只决定一件事：
系统会不会把这次明确反馈继续带出当前对话，进入后续学习和修复链路。

### 按钮

- `先暂停学习`
  - 关闭后续学习、补学、修复提案整理。
- `重新开始学习`
  - 重新允许后续学习和修复链路继续工作。

### 关闭后的效果

- 负反馈只停留在当前聊天。
- 不继续沉淀到知识库。
- 不继续整理修复提案。
- 不会再往后触发自动验证或自动应用。

### 开启后的效果

- 负反馈可以继续进入学习链路。
- 后续具体走到哪里，由 `学习强度` 决定。

## 2. 学习强度

这一块决定的是：
系统最多会继续处理到哪一步。

### 基础

用户体感：

- 会记住明显的负反馈。
- 会把经验沉淀下来。
- 不联网补学。
- 不整理修复提案。
- 不跑验证。
- 不自动落地补丁。

对应配置：

- `allow_feedback_relearn = true`
- `allow_knowledge_write = true`
- `allow_web_search = false`
- `allow_self_repair_planning = false`
- `allow_self_repair_test_run = false`
- `allow_self_repair_auto_apply = false`
- `self_repair_apply_mode = confirm`

### 进阶

用户体感：

- 会记住负反馈。
- 会联网补学相关知识。
- 会整理修复提案。
- 会先跑最小验证。
- 真正改代码前先停下来给用户看。
- 不自动落地补丁。

对应配置：

- `allow_feedback_relearn = true`
- `allow_knowledge_write = true`
- `allow_web_search = true`
- `allow_self_repair_planning = true`
- `allow_self_repair_test_run = true`
- `allow_self_repair_auto_apply = false`
- `self_repair_apply_mode = confirm`

### 深度

用户体感：

- 包含进阶档的全部能力。
- 低风险问题允许后台继续自动落地。
- 中高风险改动仍然不会直接放行。
- 中高风险最多只需要一次人工确认。

对应配置：

- `allow_feedback_relearn = true`
- `allow_knowledge_write = true`
- `allow_web_search = true`
- `allow_self_repair_planning = true`
- `allow_self_repair_test_run = true`
- `allow_self_repair_auto_apply = true`
- `self_repair_apply_mode = confirm`

### 卡片按钮

- `切到这一档`
  - 把当前配置直接改成该档预设。
- `保持这一档`
  - 当前配置已经和该档完全一致，不再改动。

### 自定义组合

如果你在 `开发者细调` 里单独改了底层开关，导致配置不再完全等于基础、进阶、深度三档中的任意一档，设置页就会显示：

- `当前是自定义组合`

这不代表坏了，只代表你现在走的是手工组合，不是标准预设。

## 3. 知识转化

这一块不是开关，而是结果看板。
它回答的问题是：

`最近这次学习，最后落成了什么结果。`

当前主要展示 4 种结果态：

- `积累经验`
  - 已经记住这次反馈，但还没进入具体修法落地。
- `生成方案`
  - 已经整理出修复提案或改法预览。
- `自动修改`
  - 已经真的动手改了，并可能通过验证或触发回滚。
- `解锁新技能`
  - 当前不是日常默认主链，通常保持关闭，不作为这套设置页的常规结果。

注意：
`知识转化` 只是展示最近结果，不是审批按钮区。

## 4. 修复提案审核

这是当前唯一保留的显式审批入口。

也就是说，现在用户不用再去记多处分散按钮，只需要记住一个地方：

`设置 -> 修复提案审核`

### 这个区块负责什么

- 看最近的修复提案。
- 看每条提案的状态、风险、验证结果。
- 看候选文件、建议测试、预览片段。
- 决定是否批准应用。

### 当前面向用户的按钮

- `刷新提案`
  - 重新从后端拉取提案列表。
- `展开详情` / `收起详情`
  - 查看或收起单条提案的详细信息。
- `生成改动预览`
  - 为这条提案生成补丁预览、文件范围和片段摘要。
- `重新生成预览`
  - 当预览失败或需要重跑时再次生成。
- `批准并应用`
  - 对这条提案做一次明确人工确认，然后执行应用。
- `开启提案整理`
  - 只会在 `allow_self_repair_planning = false` 时出现，用来重新打开提案生成链路。

### 现在没有保留的显式按钮

当前没有单独对外保留这些按钮：

- `驳回提案`
- `手动回滚`
- `再次确认弹窗`

现在的设计是：

- 不想通过，就不要点 `批准并应用`。
- 应用后如果验证失败，系统会自动回滚。
- 中高风险也统一从这一处入口确认，不再分散成多个审批面板。

## 5. 开发者细调

这一块是底层策略区，不是第二个审批中心。

### 当前底层开关

- `联网补学`
- `写入知识库`
- `负反馈纠偏补学`
- `整理修复提案`
- `先跑最小验证`
- `低风险自动落地`
- `自动生成技能`

### 当前底层参数

- `最短触发字数`
- `搜索超时（秒）`
- `结果条数上限`
- `摘要长度上限`
- `自查超时（秒）`
- `中高风险改动怎么处理`

### 这一块和审批的关系

这里决定的是“审批策略”和“修复链路的开关”，不是直接审批某一条提案。

最容易混淆的是这一项：

- `中高风险改动怎么处理`
  - `先给我看方案`
  - `只确认一次`

它们的作用是策略层面的，不是第二个审批入口。

换句话说，真正审批某条提案时，仍然回到：

`修复提案审核 -> 批准并应用`

## 修复提案的真实链路

当前真实链路如下：

`负反馈 -> 记录纠偏规则 -> 生成修复提案 -> 跑最小验证 -> 生成改动预览 -> 风险判断 -> 自动应用或等待人工确认 -> 应用后验证 -> 保留或回滚`

### 第 1 步：用户给出明显负反馈

例如用户指出：

- 回答错了
- 页面坏了
- 按钮点不动
- 路由跑偏了

这类信息会先进入纠偏规则和反馈学习链路。

### 第 2 步：是否允许整理修复提案

只有当：

- `自主学习` 开启
- `allow_self_repair_planning = true`

系统才会继续为这条反馈生成 `self_repair_report`。

如果 `allow_self_repair_planning = false`，链路会停在学习层，不会出现提案。

### 第 3 步：是否先跑最小验证

如果：

- `allow_self_repair_test_run = true`

系统会先根据提案里列出的测试路径跑一轮最小验证。

这一步的目的不是确认补丁一定正确，而是先看当前提案对应的测试环境是不是至少可跑、可判断。

### 第 4 步：生成改动预览

当前后端在修复规划任务里，会自动尝试走到这一步：

- 生成候选编辑计划
- 生成补丁摘要
- 给出拟改文件
- 给出预览片段

前端上的 `生成改动预览` 按钮，本质上是：

- 手动补跑
- 失败后重跑
- 某些情况下重新拉一次最新预览

### 第 5 步：风险判断

当前风险判断来自 `core/self_repair.py`，核心规则是：

- `高风险`
  - 预览计划生成失败，或前置验证不通过。
- `低风险`
  - 修改范围小。
  - 文件数不超过 2。
  - 编辑块数不超过 3。
  - 目标文件都在允许范围内。
  - 目标文件属于 `core/skills/` 下的相对安全路径。
- `中风险`
  - 能生成预览，也不是明显失败，但改动范围超过低风险条件。

### 第 6 步：自动应用还是人工确认

这里要分 2 层判断：

第一层：有没有开 `低风险自动落地`

- `allow_self_repair_auto_apply = true`
  - 低风险提案允许后台继续自动应用。
- `allow_self_repair_auto_apply = false`
  - 即使低风险，也会停在提案阶段。

第二层：是不是中高风险

- 中高风险不会因为开启了自动落地就直接绕过人工确认。
- 当前仍然统一在 `修复提案审核` 里通过 `批准并应用` 完成确认。

### 第 7 步：应用后验证与回滚

当补丁真正写回源码后，系统还会再跑验证。

可能出现的结果：

- `applied`
  - 应用成功，验证通过。
- `applied_without_validation`
  - 应用成功，但没有跑验证。
- `rolled_back_without_validation`
  - 应用后发现没有有效验证，自动回滚。
- `rolled_back_after_failed_validation`
  - 应用后验证失败，自动回滚。
- `rolled_back_after_apply_error`
  - 应用时报错，自动回滚。

所以当前默认思路是：

- 用户负责是否批准。
- 系统负责应用后的安全兜底。

## `先给我看方案` 和 `只确认一次` 的区别

这两个策略都不是第二个审批入口，它们都是“审批节奏”的配置。

### `先给我看方案`

含义：

- 更强调先停在方案层。
- 适合你想先看提案摘要、候选文件和预览片段时使用。

### `只确认一次`

含义：

- 中高风险改动最多保留一次人工确认。
- 不再拆成多次确认步骤。

### 当前前端的真实表现

由于现在设置页已经收口成一个显式审批区，所以这两个策略在前端上的差别，主要体现在：

- 状态文案
- 提案状态
- 是否更倾向先停在方案阶段

但最终真正的人为通过动作，仍然统一是：

- `批准并应用`

## 为什么现在只保留一个审批入口

原因有 3 个：

1. 以前入口分散时，用户很难判断哪一个才是真正生效的审批点。
2. 设置策略和审批动作混在一起，容易让人误以为“改了策略就等于已经审批”。
3. 修复提案、预览、应用本来就是一条连续链路，收成一个入口更不容易迷路。

所以现在的收口原则是：

- 策略在 `开发者细调`
- 结果在 `知识转化`
- 真正审批在 `修复提案审核`

## 现在如果只想记最关键的事

只需要记这 4 句：

1. `自主学习` 决定要不要继续学、继续修。
2. `学习强度` 决定最多走多远。
3. `知识转化` 只负责告诉你最近处理成了什么结果。
4. `修复提案审核` 是当前唯一需要记住的审批入口。

## 一句话结论

当前这套设置页已经把“学习策略”和“提案审批”重新拆清楚了：

- 学不学，看 `自主学习`
- 学多深，看 `学习强度`
- 学成什么，看 `知识转化`
- 批不批，统一到 `修复提案审核`
