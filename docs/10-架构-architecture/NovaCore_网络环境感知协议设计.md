# AaronCore 网络环境感知协议设计

> 状态：实现草案
> 目的：把“外网是否可用、是否需要代理、是否能访问目标”从隐含前提升级为显式环境协议
> 适用范围：LLM API 请求、联网搜索、远程网站访问、未来所有依赖外网的远程能力

---

## 1. 设计目标

当前系统已经有比较清晰的 6 层电脑控制底座，但“网络是否可用”还没有被当成正式环境能力来管理。

现在的问题不是“某个代理端口坏了”，而是：

- 系统默认把代理环境变量当成事实
- 没有先判断“当前环境到底能不能访问目标”
- 请求失败时，不容易区分是：
  - 无外网
  - 本地代理失效
  - 目标站点不可达
  - 模型 API 配置错误
  - 远程服务自身故障

本设计的目标是：

1. 不把固定代理端口当成系统前提
2. 只根据当前环境决定是否能访问目标
3. 访问失败时如实返回原因
4. 把网络访问统一收口到协议层，不让每个模块各自猜

---

## 2. 设计原则

### 2.1 不依赖固定端口

系统不应该默认依赖 `127.0.0.1:7890`、`7897` 等本地代理端口。

正确问题应该是：

> 当前环境能不能访问这个目标？

而不是：

> 当前本地代理端口在不在？


### 2.2 网络是环境状态，不是业务逻辑

网络能力应当被视为“环境感知层”的一部分。

先感知，再决策，再执行：

```text
感知网络环境
  ↓
判断访问路径（直连 / 代理 / 快速失败）
  ↓
执行请求
  ↓
如实返回结果或失败原因
```


### 2.3 失败必须诚实

如果用户环境无法访问目标网站，系统应明确返回：

- 当前环境无法访问该目标
- 可能是无外网 / 代理失效 / 本地网络限制

而不是伪装成功。


### 2.4 LLM 不可达与目标网站不可达必须分开

这两类错误不能混：

1. **目标网站不可达**
   - LLM 还在线
   - Agent 应该正常解释失败原因

2. **LLM API 自己不可达**
   - 连云模型都不可用
   - 应该进入系统级降级提示

---

## 3. 在当前 6 层底座中的位置

不建议额外再造“第 7 层网络层”，而是挂进现有底座：

### 第 1 层 环境感知层

新增：**网络环境感知**

负责：

- 是否存在代理环境变量
- 是否是本地代理
- 本地代理是否存活
- 是否具备基础联网能力
- 目标域名是否可达
- 默认模型 API 是否可达


### 第 2 层 文件 / 目标协议层

扩展：**远程目标协议**

本地目标：

- file
- folder
- app
- window

远程目标：

- website
- api_endpoint
- model_api

也就是说，网络目标也应该成为正式 target type，而不是临时字符串。


### 第 6 层 执行编排层

新增：**网络路由决策**

负责：

- 决定直连还是代理
- 判断是否应该快速失败
- 统一重试策略
- 统一错误分类

---

## 4. 建议新增模块

建议新增文件：

- `core/network_protocol.py`

这是一个协议层模块，不直接承担聊天主链，不直接承担 UI，只负责“网络环境 -> 路径决策 -> 失败分类”。

## 5. 推荐函数设计

## 5.1 `sense_network_environment()`

作用：

- 感知当前运行环境中的网络状态

建议返回：

```python
{
    "has_proxy_env": True,
    "proxy_env": {
        "HTTP_PROXY": "http://127.0.0.1:7890",
        "HTTPS_PROXY": "http://127.0.0.1:7890",
    },
    "has_local_proxy": True,
    "local_proxy_alive": False,
    "internet_reachable": True,
    "direct_dns_ok": True,
    "default_model_host": "api.minimaxi.com",
    "default_model_reachable": True,
    "preferred_route": "direct",
}
```

说明：

- `has_proxy_env`: 是否存在代理环境变量
- `has_local_proxy`: 是否指向 `127.0.0.1` / `localhost`
- `local_proxy_alive`: 本地代理端口是否真活着
- `internet_reachable`: 是否具备基础外网能力
- `preferred_route`: 当前建议网络路径


## 5.2 `classify_remote_target(target: str) -> dict`

作用：

- 把远程目标分成正式类型

建议返回：

```python
{
    "target_type": "website",   # website / api_endpoint / model_api / unknown
    "host": "example.com",
    "url": "https://example.com",
    "requires_external_network": True,
}
```


## 5.3 `decide_network_route(target: dict, env_status: dict) -> dict`

作用：

- 根据目标和环境状态决定怎么访问

建议返回：

```python
{
    "route": "direct",          # direct / proxy / fail_fast
    "reason": "internet_reachable_directly",
    "should_retry_without_proxy": False,
}
```

典型规则：

- 有网且能直连：`direct`
- 直连不通但代理可用：`proxy`
- 代理是本地代理但端口已死：优先 `direct`
- 没网或目标不可达：`fail_fast`


## 5.4 `format_network_failure(reason: str, target: dict, env_status: dict) -> dict`

作用：

- 统一把网络失败翻译成可读错误

建议返回：

```python
{
    "error_type": "dead_local_proxy",
    "message": "检测到本地代理环境仍存在，但代理端口不可用，已无法通过该代理访问目标。",
    "user_safe_message": "当前环境里的本地代理不可用，所以暂时访问不了这个目标。",
}
```

建议分类：

- `no_internet`
- `dead_local_proxy`
- `target_blocked_by_environment`
- `model_api_unreachable`
- `dns_failed`
- `invalid_remote_target`

## 6. 与当前代码的接入点

## 6.1 `brain/__init__.py`

这是第一优先级接入点。

原因：

- 当前所有 LLM 请求最终都走这里
- 这里是“云模型访问”最集中的地方

建议改法：

- 把当前零散的 `requests.post(...)` 收口到统一网络决策函数
- 不再仅依赖 `requests` 默认环境代理行为
- 按 `network_protocol` 的结果决定：
  - 直连
  - 走代理
  - 失败时给出明确分类


## 6.2 `routes/chat.py`

建议增加一个轻量前置检查：

- 检查当前默认模型是否可达
- 如果不可达，快速进入“云模型不可用”降级响应

作用：

- 避免用户等到超时
- 避免前端表现成“系统卡死”


## 6.3 远程技能

以下能力应逐步统一接入：

- `web_search`
- `news`
- 未来任何访问外部网站 / API 的能力

它们都不应该各自猜代理，而应该统一问协议层：

> 当前环境能不能访问这个目标？


## 6.4 `open_target`

这里需要特别注意：

- 如果目标是本地路径 / app / window：继续走本地协议
- 如果目标是远程网站：先走网络环境判定

也就是说 `open_target` 应逐步支持：

- 本地目标协议
- 远程目标协议

统一由 target classification 驱动。

---

## 7. 运行时策略

推荐统一成以下策略：

### 策略 A：能直连就直连

只要环境允许直接访问目标，就不要因为存在代理环境变量而强制走代理。


### 策略 B：代理是能力，不是前提

如果检测到代理可用，可以用；
但它不应成为系统唯一出路。


### 策略 C：本地代理失活时自动切直连

如果：

- `HTTP_PROXY/HTTPS_PROXY` 指向 `127.0.0.1`
- 本地代理端口不可用

则：

- 不要继续死走代理
- 优先尝试直连


### 策略 D：环境不允许访问时快速失败

如果当前环境本身不能访问目标，则直接返回：

- 当前环境无法访问
- 失败原因是什么

不要硬重试很多轮。

---

## 8. 前端和用户可见状态

后续建议在前端补这几类状态：

- 云模型可用
- 当前走直连
- 当前走代理
- 本地代理已失效，已自动回退直连
- 外网不可达
- 目标网站受环境限制不可访问

这能让用户明确知道：

- 是网络环境问题
- 还是模型配置问题
- 还是目标网站本身问题

---

## 9. 系统降级规则

## 9.1 当 LLM API 不可达

系统应进入“云模型降级模式”：

- 返回明确提示
- 不假装回答成功
- 仍允许本地能力继续可用（后续可扩展）


## 9.2 当目标网站不可达但 LLM 可达

系统应正常回复：

- 当前环境无法访问指定网站
- 可能原因：无外网 / 代理失效 / 环境限制

这时仍然属于“Agent 正常工作”，不是系统故障。

---

## 10. 实施顺序

建议按 3 步走：

### 第一步：先落协议，不大改主链

新增：

- `core/network_protocol.py`

先把：

- 环境感知
- 路由决策
- 错误分类

定义清楚。


### 第二步：接 LLM 请求层

把 `brain/__init__.py` 接到统一网络协议。

优先解决：

- 代理残留
- 本地代理失活
- 模型 API 不可达分类


### 第三步：接远程技能和网站访问

把：

- `web_search`
- `news`
- `open_target` 的远程网站路径

逐步统一接到 `network_protocol`。

---

## 11. 一句话结论

AaronCore 后续不该再默认“依赖某个本地代理端口”，而应该：

> **先感知当前环境是否具备访问目标的能力，再决定直连、走代理还是如实失败。**

这套协议一旦落下来，网络访问才真正成为你现有 6 层底座的一部分，而不是漂在外面的偶然行为。
