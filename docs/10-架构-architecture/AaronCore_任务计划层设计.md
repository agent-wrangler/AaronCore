# AaronCore 任务计划层设计

## 1. 目标

当前 Nova 的执行底座已经比较强：

- 能调用工具
- 能执行桌面动作
- 能做协议层验证
- 能把过程步骤显示出来

但复杂任务仍然缺一层真正的任务骨架。

比如：

- 帮我写一篇最近关于 agent 的公众号报道并发到公众号

这个任务不只是“调用几个工具”，而是天然包含：

1. 明确目标
2. 收集资料
3. 提炼结构
4. 生成内容
5. 导出或发布

所以 Nova 现在缺的是：

- `规划`
- `执行`
- `观察`
- `反思`

里的第一层和第三层之间的桥梁，也就是：

- 任务清单
- 当前阶段
- 下一步
- 阻塞点

## 2. 当前问题

目前前端显示的是 `process.steps`，它适合单步动作，但对复杂任务不够：

- 记忆加载
- 模型思考
- 调用技能
- 技能完成

这类步骤只说明“做了什么”，不说明：

- 这一步属于哪一个任务阶段
- 整个任务推进到哪里了
- 后面还剩哪些关键步骤

所以 UI 会显得：

- 不空白
- 但仍然偏“动作日志”

## 3. 新层定义

不新增“第七层”，任务计划层属于现有系统上的高层编排，不是替代 6 层协议底座。

更准确的结构应该是：

`L4 / 记忆 / 用户偏好`
-> `任务计划层`
-> `tool_call / 技能层`
-> `6 层协议底座`

其中：

- 任务计划层负责“怎么拆任务”
- 技能层负责“每一步调用什么能力”
- 协议底座负责“动作怎么可靠完成”

## 4. 计划与步骤的关系

必须明确区分两种东西：

### 4.1 任务计划

这是高层骨架，表示整件事的阶段和任务清单。

示例：

- 明确报道主题
- 收集最近 agent 资料
- 筛选和整理素材
- 生成公众号结构
- 写正文草稿
- 润色标题和结尾
- 导出或发布

### 4.2 执行步骤

这是当前每个计划项下的具体动作。

示例：

- 打开网页
- 搜索资料
- 读取文件
- 保存草稿
- 打开公众号后台

### 4.3 UI 关系

前端应当显示为：

- 上层：任务计划
- 下层：过程步骤

而不是只显示步骤。

## 6. 触发条件

不是每一轮都要先规划。

建议先用轻量判定：

### 6.1 单步任务

直接执行，不生成 plan。

例如：

- 打开豆包
- 查明天天气
- 打开百度地图

### 6.2 多步任务

生成 task plan。

例如：

- 写文章
- 做报告
- 调研并总结
- 生成内容并保存/发布
- 需要多个工具或多阶段推进的任务

### 6.3 最小判定方式

先做低风险版本：

- 用户输入含明显复杂任务词
- 或模型在首轮判断为 `multi_step`

建议不要走关键词硬路由，而是：

1. 先规则做粗过滤
2. 再让 LLM 输出是否需要 plan

输出格式例如：

```json
{
  "needs_plan": true,
  "goal": "...",
  "plan_items": [...]
}
```

## 7. 后端挂点

最稳的挂点在 `routes/chat.py`。

### 7.1 规划生成时机

建议放在：

- `记忆加载`
- `人格图谱`

之后，

- `tool_call`

之前。

也就是当前这条链：

1. 组 bundle
2. 生成或更新任务计划
3. 把计划写进 `bundle["task_plan"]`
4. 再进入 `tool_call`

### 7.2 流式事件

建议新增一个轻量事件：

- `plan`

格式例如：

```json
{
  "goal": "...",
  "phase": "research",
  "summary": "...",
  "items": [...]
}
```

前端收到后，更新当前 pending assistant message 顶部的任务骨架。

### 7.3 历史存储

当前 assistant 消息保存时：

- `process.steps` 已经能落盘

接下来只要把：

- `process.plan`

一起写进去即可。

## 8. 前端展示

### 8.1 原则

不做重型项目管理面板。

保留当前 Nova 风格：

- 简洁
- 轻
- 非卡片堆叠
- 非气泡式任务管理

### 8.2 推荐结构

在当前步骤流上方，加一个很轻的 `plan-strip`：

- 当前目标
- 当前阶段
- 任务项列表

展示方式：

```text
任务推进
明确目标  已完成
收集资料  进行中
生成结构  未开始
写正文    未开始
导出发布  未开始
```

下面保留当前步骤流：

```text
记忆加载   最近对话已接入 / 会话记忆已接入
模型思考   我先确认报道主题和时间范围
调用技能   web_search · 目标：agent 新闻
技能完成   已找到最近 8 条结果
```

### 8.3 简单任务时

如果没有 `plan`：

- UI 仍只显示原来的步骤流

这样不会把简单任务搞重。

## 9. 计划更新机制

任务计划不是一次生成后就不动。

它需要随着执行推进更新：

- 当前项完成 -> 标成 done
- 下一项开始 -> 标成 running
- 失败 -> 标成 blocked 或 error
- 需要用户决策 -> 标成 waiting_user

建议状态集合：

- `pending`
- `running`
- `done`
- `blocked`
- `waiting_user`
- `error`

## 10. ask_user 与计划层

复杂任务里最重要的一点是：

- 遇到需要用户决策时
- 不只是弹 ask_user
- 还要同步更新计划状态

例如：

- 计划项：确认写作角度
- 状态：`waiting_user`

这样前端看起来就会很顺，不像现在只是突然弹出一个选择框。

## 11. 与 L4 的关系

L4 不负责生成任务清单，但会影响任务计划的风格和取向。

例如：

- 写公众号时更偏 Nova 的口吻
- 发布前会不会更谨慎确认
- 是否优先整理再输出

所以：

- `计划层` 决定流程
- `L4` 决定流程中的表达风格与用户偏好

## 12. 第一阶段最小落地

建议不要一步做满，按三阶段推进。

### Phase 1：计划只读版

目标：

- 对复杂任务生成 plan
- 前端显示 plan
- 不做自动状态推进

价值：

- UI 立刻不空
- 用户能看到整件事骨架

### Phase 2：计划与步骤联动

目标：

- 工具执行时同步 plan 的当前阶段
- ask_user 时标记 waiting_user
- 保存历史时保留完整 plan

价值：

- 任务推进感出来

### Phase 3：计划驱动执行

目标：

- LLM 根据当前计划项决定下一步工具
- 复杂任务真正变成“按计划推进”

价值：

- 从执行型 agent 升到任务型 agent

## 13. 当前建议

最稳的下一步不是马上搞自动计划驱动执行，而是：

1. 先实现 `process.plan`
2. 先把 plan 挂到当前步骤流上方
3. 先让复杂任务有骨架
4. 再逐步把计划状态和工具执行联动起来

这样风险最小，而且最符合当前 Nova 的状态。

## 14. 对应文件

建议第一阶段主要改这几个文件：

- `routes/chat.py`
  - 生成/更新 `task_plan`
  - 发 `plan` SSE
  - 保存到 `process.plan`

- `aaron.py`
  - 在终端 TUI 中呈现 `plan` / process 事件
  - 历史与运行状态保持终端原生显示，不再依赖 Web 前端资源

## 15. 一句话总结

Nova 现在不缺动作能力，缺的是：

- 复杂任务的任务骨架

任务计划层接上之后：

- 上层看任务推进
- 下层看动作落实

这样前端过程感会自然很多，复杂任务也会更像真正的 agent，而不是“边想边试的聊天机器人”。
