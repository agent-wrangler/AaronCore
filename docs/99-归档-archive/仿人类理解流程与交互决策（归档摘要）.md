## 1. 当时想解决什么

这组方案产生于“关键词路由/轻规则分流”还很重的阶段，核心目标是减少下面这类误判：

- 用户只是聊天、讨论、举例、提供背景
- 系统却因为命中了“天气/故事/游戏/代码”之类词，直接把它当成立即执行请求

所以它本质上是在解决：

**如何让系统别因为看到技能词就过早执行。**

## 2. 第一版的核心观点

第一版更像“旧路由止血方案”，主要有 5 个点：

1. 先分 `chat / discuss / inform / task`
2. 非 `task` 不进技能匹配
3. 执行授权高于关键词命中
4. 不确定时默认不执行
5. 给技能维护反例库，压制伪命中

把它压成一句话，就是：

**话语类型识别 -> 执行授权判断 -> 技能匹配 -> 执行**

## 3. 第二版的延伸方向

第二版不是新的独立方案，更多是在第一版上继续拔高概念，提出把“意图路由”升级成“交互决策/行动治理”。

它新增强调的点主要是：

- 交互阶段识别，而不只是 task/non-task
- 响应策略选择，而不只是“调不调技能”
- 行动等级
- 最小必要行动原则
- 纠偏优先

也就是说，它已经从“防误触发”开始往“类人响应决策”发散。

## 4. 为什么现在改成一份摘要

保留原始两份长文的价值已经不高，原因主要有 4 个：

1. 两份内容高度重叠，第二版本质上是第一版的扩展表述
2. 它们都属于旧路由时代的概念稿，和当前运行态已经不对齐
3. 当前仓库的硬约束明确要求不要在 LLM 决策前再插入新的 pre-routing / pre-planning 层
4. 那个阶段确实出现过一版关键词/规则路由实施稿，但它和当前主链约束冲突，现已直接删除

## 5. 和当前 AaronCore 的关系

当前 AaronCore 的主链理解，应以代码、`RUNTIME.md`、`docs/README.md` 和 `AGENTS.md` 为准。

这组历史方案现在最多只保留两层价值：

- 作为“旧关键词路由为什么会误触发”的问题背景
- 作为“最小必要行动/纠偏优先”这类产品思路的早期草稿

但它**不应该再被当作当前架构蓝图**，尤其不应据此去：

- 在 `routes/chat.py` 主链前再补一层新路由
- 重建平行的 planning / continuity / context injection 机制
- 用关键词启发式替代现有 runtime state 锚点
