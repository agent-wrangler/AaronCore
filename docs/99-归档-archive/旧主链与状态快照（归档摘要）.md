# 旧主链与状态快照（归档摘要）

> 状态：历史快照摘要，不作为当前实现依据
> 当前运行态请优先看：`RUNTIME.md`
> 当前文档导航请优先看：`docs/README.md`

这份文档用于合并保留以下五份重复度很高、且都已经明显过时的“当前快照/阶段总结”类文档：

- `当前主链.md`
- `当前状态.md`
- `总体架构说明.md`
- `项目总览.md`
- `后续收口路线.md`

## 1. 这些文档原本在描述什么

它们描述的是 2026-03 中旬那一阶段的旧运行态快照，核心叙事基本一致：

- 桌面壳入口是 `desktop.py`
- 后端主入口是 `agent_final.py`
- 前端骨架是 `output.html` + `static/`
- 技能/路由中心仍按 `core/router.py`、`core/executor.py`、`core/skills/` 来理解
- 持久化状态主要按 `memory_db/` 组织

压成一句话，就是当时的认知模型大致是：

```text
desktop.py
  -> agent_final.py
  -> output.html / static
  -> core/router.py + core/executor.py
  -> memory_db/
```

它们同时也记录了当时的阶段性收口目标，大致包括：

- 继续把 `agent_final.py` 从单大文件往模块化方向拆
- 保持 `memory_db/` 作为统一状态目录
- 保持 `output.html` + `static/` 作为主前端形态
- 后续再评估是否拆到 `api/`、`services/`、`storage/`

## 2. 为什么现在适合合并

这五份文档的差别主要只是写法和展开层次，不是事实边界：

- `当前主链.md` 偏入口清单
- `当前状态.md` 偏项目现状罗列
- `总体架构说明.md` 偏分层讲解
- `项目总览.md` 偏总述版入口说明
- `后续收口路线.md` 偏阶段里程碑和后续计划

但它们讲的是同一阶段的旧快照，而且存在大量重复信息。

更关键的是，它们都带着“当前/总体”的名字，容易让人误以为还能代表现在的 AaronCore。

## 3. 它们为什么已经不适合作为现行说明

当前仓库对运行态文档的优先级已经很明确：

1. 代码本身
2. `RUNTIME.md`
3. `docs/README.md`
4. 其他说明文档

而这五份归档文档里的不少判断，已经和现在的真实主链不一致，例如：

- 它们还在把旧的规则路由/`agent_final.py` 时代描述成“当前主链”
- 它们没有覆盖现在以 `routes/chat.py` + tool_call 为主的运行态理解
- 它们把一些当时的目录边界和状态源写成了“当前事实”，现在只剩历史参考价值

## 4. 这组旧快照现在还剩什么价值

仍然保留一点价值，但只限于历史追溯：

- 看 AaronCore 在旧阶段是怎么理解“桌面壳 + 单体后端 + 单页前端”的
- 看旧时代如何组织入口、路由、技能和状态目录
- 帮助判断某些目录命名和历史文件为什么会存在

也就是说，它们适合回答：

**“当时是怎么想的、怎么组织的？”**

而不适合回答：

**“现在真实怎么跑？”**

## 5. 归档后的阅读建议

如果你是在追当前实现，请直接看：

- `RUNTIME.md`
- `docs/README.md`
- `AGENTS.md`

如果你是在追旧阶段的历史背景，这一组文档保留一份摘要就够了，不需要再拆成五份并排陈列。

## 6. 归档结论

对这一组文档，最合适的收口方式是：

- 保留一份“旧主链/状态快照摘要”：本文
- 删除 5 份重复的旧“当前说明/阶段总结”
- 避免继续在归档目录里堆放多个名字像“当前事实”、实际却已经失效的旧稿
