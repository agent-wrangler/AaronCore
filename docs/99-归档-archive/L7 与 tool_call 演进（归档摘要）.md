# L7 与 tool_call 演进（归档摘要）

> 状态：历史演进摘要，不作为当前实现依据
> 当前运行态请优先看：`RUNTIME.md`
> 当前文档导航请优先看：`docs/README.md`

这份文档用于合并保留以下两份历史稿：

- `L7反馈系统架构.md`
- `单步tool_call改造方案.md`

## 1. 这两份文档原本在讲什么

它们都属于 AaronCore 从旧规则主导阶段往新运行态过渡时的中间材料：

- 一份在讲：用户负反馈如何沉淀成经验、约束和修复动作
- 一份在讲：系统怎样从“规则路由 + 分离执行”迁到原生 `tool_call`

虽然主题不同，但它们都在讨论同一类问题：

**旧主链怎样演进，才能更接近现在的运行态。**

## 2. L7 旧稿的核心价值

`L7反馈系统架构.md` 最值得保留的历史点，大致有 3 个：

1. 把 L7 视作经验沉淀层，而不是知识库
2. 区分 prompt 提醒、路由约束、代码修复这几种不同强度的反馈处理
3. 强调负反馈不只是记录，还可能影响后续行为和修复流程

压成一句话就是：

**用户纠偏 -> 经验沉淀 -> 行为约束/修复升级**

## 3. tool_call 旧稿的核心价值

`单步tool_call改造方案.md` 最值得保留的历史点，大致也有 3 个：

1. 旧链路存在“路由、执行、润色”三次串行调用的开销
2. 目标是把“选工具”和“生成回复”合到同一轮 LLM 上下文里
3. executor/skill 仍然保留，但路由地位下沉，LLM 原生决策地位上升

压成一句话就是：

**从分离式 skill 路由，转向 LLM 原生 tool_call 主链**

## 4. 为什么现在不再保留原始长稿

这两份原稿现在继续单独摆着，问题主要有 4 个：

1. 它们都带有明显的阶段性迁移语气，不再是当前事实
2. 当前真实主链已经应该回到代码和 `RUNTIME.md` 去理解
3. 长稿里有不少“当时准备怎么改”的表述，现在要么已经落地，要么已经被后续实现替代
4. 归档目录里更需要的是问题摘要，而不是一堆还在讲“准备改造”的旧方案稿

## 5. 现在该怎么理解这两条线

今天再看它们，最稳的理解方式应该是：

- `L7` 这条线：看它如何把“负反馈”从一句抱怨，变成系统层面的经验约束
- `tool_call` 这条线：看系统如何从旧分流模式走向 LLM 主导

但它们都**不应该再被当作当前架构蓝图**。

如果要看当前事实：

- tool_call 主链看 `RUNTIME.md`
- 分层语义看 `docs/10-架构-architecture/8层Brain架构详解.md`
