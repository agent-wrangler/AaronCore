# NovaCore 架构优化路线图 v1

## 1. 文档目的

这份路线图用于指导 NovaCore 从“可运行原型”逐步过渡到“结构清晰、便于持续开发的本地 AI 客户端系统”。

当前目标不是大拆大改，也不是立刻增加大量新功能，而是优先完成以下几件事：

1. 收口已有架构，避免继续分叉
2. 统一路由、执行、记忆、统计等核心协议
3. 清理前端与后端之间未对齐的部分
4. 在不破坏已有功能的前提下，为后续扩展打地基

---

## 2. 当前阶段判断

NovaCore 当前已经具备以下基础：
- 本地服务可启动
- 前端聊天可用
- 技能页面、统计页面、记忆页面已有雏形
- 技能系统与自动学习系统已接入
- 具备初步的反馈回流与长期演化方向

但目前仍处于“原型完成、架构未收口”的阶段，主要问题不是功能缺失，而是：
- 同一能力存在多套实现
- 文档、代码、前端协议不完全一致
- 有些页面看起来存在，但数据和后端没有真正闭环
- 记忆与人格数据源边界不清晰

因此，第一阶段优化的核心不是“继续堆功能”，而是“先让系统说得清、跑得顺、改得动”。

---

## 3. 第一阶段总体目标

第一阶段建议控制在“小步重构，不伤现有体验”的范围内，目标如下：

### 目标 A：统一核心主链路
把 NovaCore 的主执行链路明确为：

```text
用户输入
→ agent.py 接收请求
→ router.py 决定走聊天还是技能
→ executor.py 统一执行
→ skill module 输出结果
→ think() 组织最终回复
→ memory / memory_db 按规则写入
```

### 目标 B：统一数据真源
明确以下数据各自的唯一主来源：
- persona
- history
- stats
- knowledge
- knowledge_base
- long_term

### 目标 C：统一前后端协议
确保前端页面依赖的数据结构有标准字段，不再出现“前端想要一套，后端实际给另一套”的情况。

### 目标 D：保留现有可用功能
第一阶段不追求换技术栈，不追求前后端分离，不追求引入复杂框架，只做：
- 链路收口
- 结构理顺
- 页面补齐
- 风险点兜底

---

## 4. 优化原则

整个改造过程建议遵守以下原则：

### 4.1 不大拆
当前已有功能已经能跑，说明骨架不是错的。不要为了“显得高级”而全部推倒重来。

### 4.2 一次只收一个口
先统一路由，再统一执行，再统一数据源。不要多个核心层同时重写。

### 4.3 文档与代码同步更新
每完成一个核心改动，就同步更新说明文档，避免再次出现“文档是一个系统，代码是另一个系统”的情况。

### 4.4 优先做真源，不优先做界面
如果某个问题既可以靠改 UI 掩盖，也可以靠统一数据结构根治，优先根治。

### 4.5 先保守稳定，再增强体验
先让系统不打架，再做更漂亮的前端与更高级的智能体能力。

---

## 5. 第一阶段改造顺序

建议按下面顺序推进。

---

## 5.1 第一步：统一技能主链路（最高优先级）

### 现状问题
当前 NovaCore 至少存在两套技能执行思路：
- `core/executor.py`
- `core/skills/__init__.py`

而 `agent.py` 实际主要调用的是 `core.skills.execute_skill()`，导致：
- 文档里写的执行器不一定是真实主入口
- 新技能接入标准不统一
- 路由与执行之间的结构越来越容易分叉

### 改造目标
明确唯一技能主链路为：

```text
router.py -> executor.py -> core/skills/*.py
```

### 具体动作
1. 保留 `core/skills/` 作为技能实现目录
2. 保留 `core/skills/*.json` 作为技能元数据
3. 让 `core/executor.py` 成为唯一执行总入口
4. `agent.py` 不再直接调用 `core.skills.execute_skill()`
5. `core/skills/__init__.py` 只负责“扫描、加载、注册”，不再承担主执行职责

### 改完后的收益
- 技能架构终于收口
- 新技能新增方式变清晰
- 路由、执行、技能三层职责明确
- 后续接日志、任务状态、失败重试都更容易

---

## 5.2 第二步：统一路由真源

### 现状问题
目前既有 `core/router.py`，又有 `agent.py` 内嵌的意图判断和关键词匹配逻辑。

这意味着：
- 路由规则可能在两个地方漂移
- 改一个地方，另一个地方可能不生效
- 调试时很难判断真正使用的是哪套规则

### 改造目标
把所有“是否调用技能、调用哪个技能、为什么调用”的逻辑统一收口到 `core/router.py`。

### 具体动作
1. `agent.py` 只负责接收请求和组织返回
2. `router.py` 输出标准结构，例如：

```json
{
  "mode": "chat | skill | fallback",
  "skill": "weather",
  "confidence": 0.92,
  "reason": "命中天气关键词"
}
```

3. 不再在 `agent.py` 重复写一套关键词匹配
4. 把现有的 LLM 意图判断改为：
   - 可选增强
   - 不作为第二套独立主链路

### 改完后的收益
- 所有路由决策都有唯一来源
- 以后做路由调试面板会很容易
- 文档可以真正写清楚而不是“理想流程”

---

## 5.3 第三步：统一 memory / memory_db 的职责

### 现状问题
当前 `memory/` 和 `memory_db/` 中存在重复文件和边界模糊问题。

这会导致：
- 数据主源不清楚
- 同步规则不清楚
- 前端不知道该读哪个目录

### 改造目标
明确：

#### `memory/`
只存运行期数据：
- 当前会话缓存
- 临时统计
- 临时界面状态（如果以后有）

#### `memory_db/`
只存持久化数据：
- persona
- long_term
- knowledge
- knowledge_base
- evolution
- topic_summary

### 具体动作
1. 列出现有重复文件清单
2. 为每个文件指定唯一真源
3. 前端接口只读真源，不做双目录混读
4. 暂时保留兼容层，避免一次迁移把旧功能打断
5. 第二阶段再考虑彻底迁移或清理历史冗余

### 改完后的收益
- 数据关系变清楚
- 记忆系统更容易持续增强
- 以后做导出、备份、恢复会更稳

---

## 5.4 第四步：统一 stats 数据结构

### 现状问题
前端统计页需要的字段，比后端接口实际返回的字段更多，导致有些卡片只是样子在，数据并不完整。

### 改造目标
定义 NovaCore 统计接口统一结构，例如：

```json
{
  "stats": {
    "input_tokens": 0,
    "output_tokens": 0,
    "total_tokens": 0,
    "total_requests": 0,
    "cache_hit": 0,
    "model": "MiniMax-M2.5",
    "last_used": "2026-03-12 14:00"
  }
}
```

### 具体动作
1. 后端先按统一结构输出
2. 前端统一按 `stats.xxx` 读取
3. 当前拿不到的数据可以先明确填 0，不要返回缺字段
4. 后续再逐步补真实统计逻辑

### 改完后的收益
- 统计页稳定
- 前端代码更干净
- 后续接趋势图或细分统计更容易

---

## 5.5 第五步：统一 persona 数据源

### 现状问题
当前 `nova_name` 接口读取路径和实际人格数据主存储路径不一致。

### 改造目标
明确 `memory_db/persona.json` 为人格真源。

### 具体动作
1. `get_nova_name()` 统一从真源读取
2. 后续所有人格相关更新也都写回同一处
3. 若存在旧路径，保留兼容读取但不再作为主路径

### 改完后的收益
- 前端显示的名字和人格设置一致
- 人格能力后续更容易扩展为“设置页”

---

## 5.6 第六步：清理前端残留与半接入页面

### 现状问题
前端中存在：
- 已写但未完整挂载的页面逻辑
- 旧菜单残留
- 部分页面结构已存在，但未形成正式导航入口

### 改造目标
让前端结构和当前功能完全一致，不再出现“代码里有第5页、第6页，界面里却没有正式入口”的情况。

### 具体动作
1. 清理 `show(n)` 中无效分支
2. 把“历史页”正式决定为：
   - 要么进入主菜单
   - 要么合并进记忆页
3. 确保所有导航项都有后端支持和页面内容闭环
4. 把页面结构注释清楚，降低后续修改门槛

### 改完后的收益
- 前端状态管理更干净
- 后续继续扩展页面时不容易乱
- 产品感更强

---

## 5.7 第七步：为高风险能力加保护层

### 现状问题
例如自动生成并运行 Python 代码的能力，目前更像“实验能力”，风险较高。

### 风险点
- 模型生成代码不稳定
- 可能弹异常窗口
- 可能写入不受控路径
- 失败时不易理解原因

### 改造目标
不给这个能力下线，但要给它加基本保护。

### 具体动作
1. 增加执行前确认（前端确认或系统确认）
2. 指定临时目录作为唯一运行目录
3. 加超时和失败日志
4. 返回结构化失败结果，而不是只给一句“启动失败”

### 改完后的收益
- 降低误伤和误判
- 保留原型亮点能力
- 为以后做任务型技能打基础

---

## 6. 第一阶段建议不要做的事

为了避免重构失控，第一阶段不建议碰这些：

### 6.1 不做前后端彻底分离
现在前端虽然内嵌在 `agent.py` 中，但还能用。第一阶段先别为了“现代化”而拆 React / Vue。

### 6.2 不重写整个记忆系统
先统一真源和职责，别一上来就推翻全部 memory 结构。

### 6.3 不急着上多会话
多会话很重要，但它属于第二阶段增强项，不适合在主链路未收口时先做。

### 6.4 不急着做复杂工作流编排
当前连执行总线都没完全统一，过早做任务编排会把复杂度翻倍。

---

## 7. 改造完成后的阶段成果

如果第一阶段按顺序做完，NovaCore 会得到这些明确收益：

### 工程层面
- 核心链路明确
- 技能架构统一
- 数据真源清晰
- 接口协议稳定
- 页面结构更一致

### 产品层面
- 聊天、技能、记忆、统计形成更完整闭环
- UI 不再只是“看起来有”，而是每块都有真实数据支撑
- 人格与记忆的存在感更强

### 后续扩展层面
- 更适合继续加设置页
- 更适合继续加多会话
- 更适合继续加任务中心 / 调试面板 / 自动学习审查页

---

## 8. 第二阶段建议方向（预览）

第一阶段完成后，再进入第二阶段增强。方向建议如下：

1. 设置中心
2. 正式历史页 / 会话页
3. 真正的记忆检索与编辑
4. 技能执行状态面板
5. 自动学习结果审查面板
6. 路由决策可视化
7. 多会话与会话摘要

这些能力不需要现在立刻做，但第一阶段的收口，就是为了让这些能力到时候能顺利长出来。

---

## 9. 推荐下一步行动

基于当前情况，建议下一步不要再继续讨论抽象概念，而是直接开始执行第一阶段的第一个任务：

> **任务 1：统一技能主链路与执行入口**

这是整个系统最关键的一步。只要这一步落地，后面的路由统一、日志补齐、前端状态增强都会顺很多。

建议在正式动手前，再额外补一份简短的实施清单：
- 哪些文件要改
- 每个文件改什么
- 哪些旧逻辑先保留兼容
- 如何验证改动没有破坏现有功能

这样就能进入真正稳定的优化阶段，而不是继续“边长边乱”。
