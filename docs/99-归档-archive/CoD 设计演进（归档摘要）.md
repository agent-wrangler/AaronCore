## 1. CoD 当时想解决什么

这条线的核心问题其实很简单：

**不要把 L1-L8 一股脑塞进 prompt。**

它想解决的是旧阶段的 prompt 臃肿、上下文预加载浪费、轻闲聊也要背着整包记忆出门的问题。

## 2. 这两份文档各自提供了什么

- `CoD-Context-on-Demand.md`
  - 很短，像一张概念卡片
  - 核心是 `Flash / Trace Back` 这组二分法

- `Context-on-Demand (CoD)（按需上下文系统）.md`
  - 更长，更像一份对话式设计展开
  - 核心是“延迟挂载 / 按需拉取 / 工具化检索”

两份放在一起看，本质上讲的是同一件事：

**默认只带最小必要上下文，其余记忆在需要时再取。**

## 3. 这条设计线最值得保留的历史观点

把长文压缩后，CoD 的历史设计核心大致有 4 点：

1. 默认最小上下文，而不是全量预装
2. 把深层记忆/知识/技能封装成可调用入口，而不是每轮硬塞
3. 让模型在需要时自己拉取额外信息
4. 把“当前能直接回答”和“需要追溯历史/知识”分开处理

也可以用旧文档里的说法概括为：

- `Flash`：当前上下文足够，直接响应
- `Trace Back`：当前上下文不够，再去深层记忆/知识里找

## 5. 和当前 AaronCore 的关系

今天再看 CoD，最稳的理解方式应该是：

- 它不是一个应该继续独立膨胀的新 pre-routing 层
- 它也不该被理解成“在 LLM 前再补一层规则脑”
- 它更接近一种**按需上下文装载原则**

这点和仓库约束是对齐的：

- 不要在 LLM 决策前插入新层
- 不要重建平行的 routing / planning / continuity 机制

所以这条历史设计线今天最多保留为：

- “为什么会走向最小上下文装载”的背景说明
- “为什么要避免全量预加载”的旧问题记录
