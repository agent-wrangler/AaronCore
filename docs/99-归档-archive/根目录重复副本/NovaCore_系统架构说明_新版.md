# NovaCore 系统架构说明（新版）

## 1. 项目定位

NovaCore 是一个运行在本地的 AI 智能体客户端与服务系统，部署路径为：

`C:\Users\36459\NovaCore`

它以 **FastAPI + 本地前端页面** 作为交互入口，监听 **8090 端口**，提供聊天、技能调用、记忆展示、统计查看等能力。

NovaCore 在运行形态上是**独立进程**，可以单独启动；但在能力层面会与 OpenClaw 生态发生协同，例如调用本地记忆接口或共享部分模型 / 周边能力。因此更准确的定位是：

> **独立运行的本地 AI 客户端，带有部分 OpenClaw 协同能力。**

---

## 2. 总体架构概览

NovaCore 当前采用“**8层 Brain 设计框架 + 技能执行引擎 + 本地记忆存储**”的整体架构。

核心流程如下：

```text
用户输入
  ↓
API入口（agent.py / FastAPI）
  ↓
L1-L4 轻量理解与上下文拼接
  ↓
L5 技能路由决策
  ↓
L6 技能执行
  ↓
LLM润色回复 / 普通对话回复
  ↓
L7 反馈沉淀
  ↓
L8 自动学习与知识补充
  ↓
写入 memory / memory_db
```

需要注意的是，当前版本并不是严格的“全层强约束流水线”，而是：

> **以 L5 路由、L6 执行、L8 学习为核心落地点，结合 LLM 对话能力形成的混合式智能体架构。**

## 3. 8层 Brain 设计说明

### L1：记忆粒子
负责保留原始对话片段与近期输入输出，是最靠近用户原始消息的一层。

**主要载体**
- `memory/msg_history.json`
- `memory_db/msg_history.json`

**作用**
- 保存最近对话
- 为后续上下文理解提供基础材料

### L2：记忆提炼
负责从原始对话中提炼短期上下文，形成“当前正在聊什么”的轻量理解。

**当前形态**
- 主要通过聊天历史读取和上下文拼接实现
- 还没有完全独立成强模块

### L3：记忆结晶
负责将值得长期保留的信息沉淀为长期记忆。

**主要文件**
- `memory_db/long_term.json`

**典型内容**
- 用户的重要偏好
- 被明确要求记住的事实
- 后续可复用的稳定信息

### L4：人格图谱
负责保存 NovaCore 与用户之间的角色信息、称呼偏好、人格设定。

**主要文件**
- `memory_db/persona.json`

**作用**
- 决定称呼方式
- 影响回复语气
- 支持“叫我 xxx”“记住 xxx”这类人格或偏好更新

### L5：技能矩阵 / 路由层
这是 NovaCore 当前最核心的一层之一，负责：

- 分析用户意图
- 判断是否需要调用技能
- 从知识配置中找到匹配路由
- 决定走“技能执行”还是“普通对话”

**主要文件**
- `core/router.py`
- `memory_db/knowledge.json`

**当前实现特点**
- 有意图解析逻辑
- 有关键词触发逻辑
- 实际运行中，**关键词匹配仍然是主要触发依据**
- 属于“规则路由 + 轻量理解”的混合模式

### L6：技能执行层
这一层负责真正调用能力函数，得到结果。

**主要文件**
- `core/executor.py`
- `core/skills/*.py`
- `core/skills/*.json`
- `core/skills_loader.py`

**当前已见技能能力**
- 天气查询
- 讲故事
- 画图
- 股票查询
- 代码生成 / 运行

**执行机制**
- 根据技能名或执行函数名查找注册函数
- 调用函数执行
- 返回统一结构结果
- 再交给上层组织最终回复

### L7：经验沉淀 / 反馈层
负责处理负反馈、自检、失败回流。

**典型触发**
- 用户说“不对”
- 用户说“错了”
- 用户说“不喜欢”“重来”等

**当前作用**
- 检测负面反馈
- 找到上一轮问题
- 触发 L8 自动学习或纠偏逻辑

**相关数据**
- `memory_db/evolution.json`

### L8：能力进化 / 自动学习层
这是 NovaCore 当前另一块非常关键的扩展层。

**主要文件**
- `core/l8_learn.py`
- `memory_db/knowledge_base.json`

**职责**
- 在回答失败或知识不足时进行自动搜索
- 将学到的新知识沉淀下来
- 为后续技能增强或回复改进提供依据

**当前外部能力来源**
- Bing RSS 搜索
- Open-Meteo 天气接口
- 其他在线查询接口

**触发特点**
- 不是所有输入都学习
- 有白名单 / 长度 / 疑问特征等过滤规则
- 避免被“你好”“哈哈”这类消息误触发

## 4. 实际运行流程

NovaCore 当前真实运行流程可以概括为下面这条主链路：

### 4.1 聊天请求入口
用户请求由 `agent.py` 的 `/chat` 接口接收。

处理内容包括：
- 记录用户输入
- 写入历史
- 检查是否存在负反馈
- 决定下一步走技能还是普通对话

### 4.2 负反馈优先处理
如果用户输入命中：
- 不对
- 错了
- 不是
- 不好用
- 重来

系统会优先将其视为 **L7 反馈事件**，并尝试：
1. 找到上一轮问题
2. 触发 L8 自动学习
3. 用新的学习结果重新生成答复

这让 NovaCore 具备一种简单的“说错后补学”的能力。

### 4.3 技能识别与调用
如果不是反馈纠偏场景，则进入技能判断。

当前实现里，虽然存在“LLM 先理解意图”的逻辑，但**最终真正稳定生效的还是关键词匹配优先**：

1. 读取所有可用技能
2. 检查用户输入中是否命中技能关键词
3. 若命中，则调用对应技能函数
4. 得到技能结果后，再交给 `think()` 做语言润色

### 4.4 普通对话模式
如果没有匹配到明确技能，则进入普通对话模式：

- 将用户问题与可用技能信息一起提供给思考层
- 由 LLM 自行组织回复
- 输出最终自然语言结果

因此 NovaCore 当前本质上是：

> **技能优先的聊天系统，而不是纯聊天系统外挂技能。**

## 7. 技能系统架构

NovaCore 的技能系统建议统一分为三部分理解：

### 7.1 技能定义
保存在：
- `core/skills/*.json`
- `memory_db/knowledge.json`

主要定义：
- 技能名称
- 描述
- 关键词
- 触发方式
- 执行函数映射

### 7.2 技能加载
由：
- `core/skills_loader.py`

负责：
- 扫描技能目录
- 动态导入技能模块
- 注册到执行器函数表

### 7.3 技能执行
由：
- `core/executor.py`

负责：
- 根据 exec_func 或技能名找到函数
- 执行函数
- 返回统一结果结构

当前技能系统已经具备基本扩展能力，但仍有几个明显特点：
- 路由和执行之间的协议还不够统一
- 有些地方写的是技能名，有些地方写的是执行函数名
- 存在“配置驱动”和“硬编码驱动”混用现象

这意味着后续如果要继续升级，最值得先整理的是：

> **统一技能元数据格式、统一路由字段、统一执行返回结构。**

### 8.1 运行期记忆

特点：

位于 `memory_db/`

用于保存：
- 技能知识

- 演化记录
- 搜索学习结果

特点：
- 偏长期
- 偏结构化
- 偏可复用

## 10. 核心架构基石原则

以下原则是 NovaCore 后续开发、修复、重构时必须遵守的架构基石。

### 基石原则 1：技能结果提供内容骨架，润色层提供人格表达。

解释：
- 技能层负责真实结果、事实内容、执行状态。
- 润色层负责语气、人格、温度、风格适配。
- 技能层输出的是“内容骨架”，润色层输出的是“人格表达”。

禁止：
- 润色层脱离技能结果重新发明答案。
- 润色层覆盖或篡改技能事实。

允许：
- 在不改变事实的前提下，把技能结果说得更自然、更贴近用户偏好。

### 基石原则 2：是否触发技能，不看语气像不像聊天，而看是否需要调用某种能力才能满足用户真正意图。

解释：
- 语气轻松、像闲聊，不代表它不是技能请求。
- 如果不调用某种能力，就无法完成用户意图，那么它就是技能型请求。

例如：
- “今天天气怎么样呀” 虽然很口语化，但仍然需要天气技能。
- “知道我是谁吗” 不需要外部技能，更偏向聊天与记忆。

### 基石原则 3：路由结果不应强行二分，而应支持 chat / skill / hybrid 三种结果。

解释：
- `chat`：纯聊天
- `skill`：纯技能
- `hybrid`：技能执行 + 情绪 / 人格表达并行

很多真实用户输入既有任务需求，也有聊天语气，因此 hybrid 是必须存在的。

### 基石原则 4：只要存在明确或合理的技能候选，系统不得直接将请求降为纯聊天回答。

解释：
- 只要用户输入已经表现出明确或合理的技能候选，系统就必须优先进入 skill 或 hybrid 路由。
- 不能因为语气像聊天、表达比较委婉，或者 LLM 觉得“我可以直接答”，就跳过技能层。

例如：
- “想听一个更治愈的故事” → 属于 story 技能候选
- “帮我画个温柔一点的图” → 属于 draw 技能候选
- “今天天气怎么样呀” → 属于 weather 技能候选

### 基石原则 5：LLM 可以辅助理解表达需求，但不能替代路由层决定是否跳过技能。

解释：
- LLM 可以帮助理解语气、情绪、风格偏好。
- 但是否需要调用技能，必须优先由路由层和技能候选规则决定。
- LLM 不能因为“看起来可以直接聊”就替系统跳过技能。

推荐参考文档：
- `NovaCore_路由判断规则说明.md`

## 11. 当前系统的真实特点

结合代码与文档，NovaCore 当前最真实的系统特点是：

### 优点
1. **已经具备完整可运行闭环**
   - 能启动
   - 能聊天
   - 能展示技能和记忆
   - 能调用外部接口

2. **技能化方向明确**
   - 已经有路由、执行器、技能目录
   - 后续适合继续做成插件式结构

3. **有反馈回流意识**
   - 不是只答一次
   - 有 L7/L8 的纠错与补学思路

4. **前端体验已具雏形**
   - 不只是 API
   - 已经是完整客户端原型

### 当前问题
1. **文档和代码不完全一致**
2. **“独立 / 依赖 OpenClaw”表述冲突**
3. **L5 路由与 L6 执行协议不够统一**
4. **memory 和 memory_db 职责边界不够严格**
5. **前端内嵌于 agent.py，后续维护成本会升高**
6. **8层架构部分层级仍偏概念化，需继续落地**
