# NovaCore 任务1：统一技能主链路与执行入口（实施清单）

## 1. 任务目标

本任务的目标不是增加新功能，而是把 NovaCore 当前分叉的技能调用链路收口，统一成一条稳定、清晰、可维护的主链路。

统一后的目标结构如下：

```text
用户输入
→ agent.py 接收请求
→ router.py 决定 chat / skill
→ executor.py 统一执行
→ core/skills/*.py 返回结果
→ think() 组织最终回复
→ 写入 history / memory / memory_db
```

本次任务完成后，需要达到以下结果：

1. `agent.py` 不再直接承担技能匹配与技能执行职责
2. `core/router.py` 成为唯一的路由真源
3. `core/executor.py` 成为唯一的执行真源
4. `core/skills/` 目录只负责技能实现与元数据
5. 技能新增、调试、扩展的方式统一

---

## 2. 当前问题回顾

当前项目中技能相关能力至少分散在以下几个位置：

- `agent.py`
- `core/router.py`
- `core/executor.py`
- `core/skills/__init__.py`
- `core/skills/*.py`
- `core/skills/*.json`

主要问题如下：

### 2.1 `agent.py` 中混入了路由与执行逻辑
当前 `agent.py` 不只是接 API 请求，还额外做了：
- 技能列表读取
- 关键词匹配
- 技能执行
- LLM 意图理解

这导致主入口文件职责过重。

### 2.2 `core/router.py` 存在，但未成为唯一真源
虽然已经有单独的路由模块，但主流程没有完全走它，造成实际运行逻辑和设计文档不一致。

### 2.3 `core/executor.py` 存在，但未成为唯一执行入口
当前执行器文件中定义了执行函数和执行接口，但真实主流程更多走的是 `core.skills.execute_skill()`。

### 2.4 `core/skills/__init__.py` 同时承担了加载与执行职责
技能包初始化文件现在不仅加载技能，还包含执行逻辑，容易与执行器职责冲突。

---

## 3. 改造后的职责划分

### 3.1 `agent.py`
只负责：
- 接收 HTTP 请求
- 维护聊天主流程
- 调用路由
- 调用执行器
- 组织最终回复
- 写入历史与统计

不再负责：
- 技能关键词匹配
- 技能执行细节
- 多套路由判断

### 3.2 `core/router.py`
只负责：
- 分析用户输入
- 决定是聊天还是技能
- 返回标准路由结果

建议输出结构：

```json
{
  "mode": "chat | skill | fallback",
  "skill": "weather",
  "confidence": 0.95,
  "reason": "命中天气关键词",
  "params": {}
}
```

### 3.3 `core/executor.py`
只负责：
- 根据路由结果执行技能
- 对执行异常进行统一封装
- 返回统一执行结果

建议输出结构：

```json
{
  "success": true,
  "skill": "weather",
  "response": "今天天气晴，22°C",
  "error": null
}
```

### 3.4 `core/skills/__init__.py`
只负责：
- 自动扫描技能目录
- 加载技能模块与元数据
- 提供技能注册表查询能力

不再负责：
- 作为主执行入口

### 3.5 `core/skills/*.py`
只负责：
- 具体技能功能实现
- 提供统一命名的执行函数（建议统一为 `execute(user_input)`）

### 3.6 `core/skills/*.json`
只负责：
- 技能名称
- 描述
- 关键词
- 分类
- 优先级
- 可选执行配置

## 4. 具体修改文件清单

下面按“必须修改 / 建议修改 / 暂时不动”来分。

---

## 4.1 必须修改文件

### 文件 1：`agent.py`

#### 需要改的点
1. 删除或下沉内部关键词匹配逻辑
2. 删除或弱化内部“第二套路由”逻辑
3. 不再直接调用 `core.skills.execute_skill()`
4. 改为：
   - 调用 `router.route(msg)` 获取路由结果
   - 如果是 `chat`，走普通回复
   - 如果是 `skill`，调用 `executor.execute(route_result, msg)`
5. 保留：
   - 负反馈处理
   - 历史记录
   - think() 润色
   - stats 更新

#### 改后效果
`agent.py` 从“什么都干”变成“主流程调度器”。

---

### 文件 2：`core/router.py`

#### 需要改的点
1. 统一返回结构，不再返回松散结构
2. 明确 `mode` 字段：`chat / skill / fallback`
3. 根据技能注册表或知识配置统一匹配技能
4. 保留意图判断，但不要再让它和主流程分裂
5. 返回内容至少包含：
   - mode
   - skill
   - confidence
   - reason
   - params

#### 改后效果
所有路由判断可从这里单点排查。

---

### 文件 3：`core/executor.py`

#### 需要改的点
1. 去掉与当前技能加载体系冲突的重复执行逻辑
2. 不再维护一套与 `core/skills/` 平行的硬编码主体系
3. 改成统一从技能注册表拿技能并执行
4. 统一返回结构：
   - success
   - skill
   - response
   - error

#### 改后效果
执行器成为唯一执行总线，后续加日志、任务状态、失败重试都能挂这里。

---

### 文件 4：`core/skills/__init__.py`

#### 需要改的点
1. 保留技能扫描、加载、注册能力
2. 移除“主执行入口”定位
3. 保留：
   - `load_skills()`
   - `get_skill()`
   - `get_all_skills()`
4. 如有必要，可保留底层 `execute_skill()` 作为内部兼容层，但不再让 `agent.py` 直接依赖它

#### 改后效果
技能包回归“注册中心”职责，而不是“注册中心 + 执行器”双重角色。

---

## 4.2 建议顺手修改文件

### 文件 5：`core/skills/*.py`

#### 需要检查的点
- 是否都提供统一的 `execute(user_input)` 接口
- 返回值是否尽量统一为字符串或结构化字符串
- 是否存在异常未处理情况

#### 改后效果
技能层接口统一，执行器逻辑更简单。

---

### 文件 6：`core/skills/*.json`

#### 需要检查的点
- 关键词字段是否统一使用 `keywords`
- 是否有缺失 `name / description / priority / category`
- 是否需要为后续路由加入 `aliases / tags`

#### 改后效果
技能元数据更规范，技能页和路由层都更稳定。

---

## 5. 推荐改造步骤（执行顺序）

### 第 1 步：定义标准返回结构
先不要直接重写代码，先在脑中把两套结构统一。

#### 路由统一结构
```json
{
  "mode": "chat | skill | fallback",
  "skill": "weather",
  "confidence": 0.95,
  "reason": "命中关键词: 天气",
  "params": {}
}
```

#### 执行统一结构
```json
{
  "success": true,
  "skill": "weather",
  "response": "今天天气晴，22°C",
  "error": null
}
```

---

### 第 2 步：先改 `core/router.py`
原因：先统一“怎么选技能”，再统一“怎么执行技能”。

动作：
1. 确保 `route(text)` 返回标准结构
2. 支持匹配到技能时返回 `mode=skill`
3. 匹配不到时返回 `mode=chat`
4. 暂时不引入复杂 LLM 判断，先保证规则稳定

---

### 第 3 步：再改 `core/executor.py`
动作：
1. 接受 `route_result` 和 `user_input`
2. 根据 `route_result.skill` 从技能注册表拿技能
3. 执行并统一返回结构
4. 不再让执行器依赖另一套平行硬编码主流程

---

### 第 4 步：调整 `core/skills/__init__.py`
动作：
1. 保留加载注册能力
2. 把执行相关责任降级为“兼容层”
3. 明确其角色是技能注册中心

---

### 第 5 步：最后改 `agent.py`
动作：
1. 接收用户消息
2. 做 L7 负反馈检测（保留现状）
3. 调用 `router.route(msg)`
4. 若为技能则调 `executor.execute(route_result, msg)`
5. 若为聊天则走 `think()`
6. 技能结果仍可交给 `think()` 润色
7. 写历史、写 stats

原因：
- 把主入口留到最后改，可以最大程度减少中途系统不可用的风险。

---

## 6. 兼容策略

为了避免一次改动把现有功能搞崩，建议保留一层短期兼容策略。

### 6.1 保留旧函数，但不再作为主入口
例如：
- `core/skills.__init__.py` 中如果保留 `execute_skill()`，可以继续存在
- 但 `agent.py` 不再直接依赖它

### 6.2 技能元数据保持兼容
如果已有 `.json` 字段不完全统一，第一步不要大改格式，可以先在加载层做兼容读取。

### 6.3 先兼容字符串结果
当前技能多数直接返回字符串。第一阶段不强制全部技能返回对象，执行器可先统一封装成标准结构。

## 7. 验证清单（每改完一步都要测）

本任务每完成一层，都建议做一次快速人工验证。

### 验证 1：聊天是否仍能正常回复
测试语句：
- 你好
- 你是谁
- 哈哈

预期：
- 不触发技能
- 正常走聊天回复

### 验证 2：天气技能是否还能触发
测试语句：
- 常州天气怎么样
- 上海明天气温多少

预期：
- 路由命中天气技能
- 执行器执行天气模块
- 最终返回正常结果

### 验证 3：故事技能是否还能触发
测试语句：
- 讲个故事

预期：
- 路由命中 story
- 正常返回故事内容

### 验证 4：代码技能是否仍可走通
测试语句：
- 做个猜数字游戏

预期：
- 能命中 run_code
- 至少返回清晰结果或清晰错误信息

### 验证 5：技能页是否仍正常显示
预期：
- `/skills` 返回不受影响
- 前端技能页继续能展示已注册技能

### 验证 6：负反馈流程不受破坏
测试语句：
- 不对
- 错了

预期：
- 仍能进入 L7/L8 分支
- 不会因为技能链路重构而消失

---

## 8. 完成本任务后的直接收益

如果任务 1 完成，NovaCore 会立刻获得这些好处：

1. 技能系统终于有唯一主链路
2. 以后加技能不再需要猜“该接哪一层”
3. 路由和执行更容易调试
4. 文档终于可以与代码一致
5. 后续做日志、状态、重试、任务中心时更容易落地

## 10. 建议的实际执行方式

如果接下来正式动手，建议采用以下方式：

### 方式
- 一次只改一个核心文件
- 每改一步都立即手测
- 能兼容就先兼容，不搞一次性大迁移
- 改完同步更新文档

### 推荐顺序
1. `core/router.py`
2. `core/executor.py`
3. `core/skills/__init__.py`
4. `agent.py`
5. 手测验证
6. 更新系统说明文档

这个顺序最稳，也最符合当前 NovaCore 的状态。
