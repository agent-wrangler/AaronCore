# 路由优化方案

> 最后更新：2026-03-20 21:40
> 给新对话的上下文文件。包含：当前路由架构、已完成的改动、遗留问题、下一步方案。

## 一、路由架构概览（当前）

路由决定用户消息走"LLM 聊天"还是"技能执行"。核心原则：**规则层快筛 + LLM 裁决灰色地带**。规则层只在高置信度（>= 0.9）时独立决策，中间地带交给 LLM 判断。

```
用户输入
  │
  ├─ 阶段 0：硬编码上下文规则（故事追问、新闻缺失检测）
  │     → 命中则直接返回，不走后续
  │
  ├─ 阶段 0.5：交互阶段门控
  │     → 请求阶段 + 指令态 → 强制走技能，不给 LLM 聊天机会
  │
  ├─ 阶段 1：关键词路由 core/router.py → route()
  │     ├─ 特殊检测层（笑话、股票、纠偏、文章选择等 _looks_like_xxx）
  │     │     → 命中则直接返回，不受意图分类影响
  │     ├─ _score_text() 计算 chat_score / skill_score（含反例库 + L7 约束过滤）
  │     ├─ classify_stage() 6 阶段交互识别（v2）
  │     │     → correct/confirm/inform/explore/request/social
  │     │     → 映射为旧 intent：task/discuss/inform/chat
  │     │     → discuss/inform/chat → 直接走 chat，跳过技能匹配
  │     │     → task → 允许进入技能匹配
  │     └─ 技能候选层：根据关键词具体程度分级置信度
  │           → confidence >= 0.9 直接走，不问 LLM
  │           → confidence 0.5-0.9 → 交给 LLM 裁决
  │
  ├─ 阶段 1→2 过渡：意图门控 route_resolver.py
  │     → confidence >= 0.9 → 直接返回，不问 LLM
  │     → discuss/inform 且 conf >= 0.9 → 直接返回（只有高置信度才短路）
  │     → 其余 → 交给 LLM lite 裁决
  │
  ├─ 阶段 2：轻量 LLM 裁决 route_resolver.py → llm_route_lite()
  │     → ~200 token prompt，带 L2 场景摘要 + 可用技能列表 + few-shot 示例
  │     → 如果阶段 1 有候选（confidence >= 0.7），作为 hint 传给 LLM
  │     → LLM 返回 {"intent": "chat 或技能名", "reason": "..."}
  │     → LLM 说 skill → 走技能；LLM 说 chat → 走聊天（LLM 有最终判断权）
  │
  └─ 阶段 3：兜底（保守回退）
        → lite 失败时：只有 confidence >= 0.85 才信关键词结果
        → 否则默认走 chat（不确定时宁可不执行）
        → 都没有则走重量级 LLM 路由（完整 L1-L8 上下文）
```

涉及文件：
- `core/router.py` — 关键词评分 + 意图分类 + 特殊检测（阶段 1）
- `core/route_resolver.py` — 多阶段编排 + LLM 路由 + 意图门控（阶段 0/2/3）
- `core/skills/__init__.py` — 技能注册（含 anti_keywords 加载）
- `core/skills/*.json` — 各技能元数据（含 anti_keywords 反例库）
- `agent_final.py` — POST /chat 端点，调用 resolve_route()

## 二、踩过的坑（debug 记录）

1. **"常州天气怎么样"回退到 chat**：classify_intent 加了组合证据门槛后，这个输入只有 1 项证据（技能关键词），被判为 chat。修复：多个技能关键词命中（skill_score >= 2.0）直接算 evidence = 2。
2. **"如果要画海报的话"误触发 draw**：discuss 检测被绕过（同时有任务词"画"+技能关键词"海报"）。修复：新增 STRONG_HYPOTHETICAL（如果/假如/万一），无条件返回 discuss。
3. **LLM lite 翻盘 discuss 分类**："头脑风暴吧 或者玩游戏都可以" router 返回 chat/discuss（conf 0.85），但 0.85 < 0.9 进了 LLM lite，LLM 返回 story。修复：在 resolve_route 加意图门控，discuss/inform + conf >= 0.75 直接返回。**后续问题**：0.75 门控过于激进，把"天气怎么样"等合法技能请求也拦了（见第三轮 3.1）。
4. **anti_keywords 不生效**：`core/skills/__init__.py` 注册时没提取 anti_keywords 字段。修复：加 `'anti_keywords': meta.get('anti_keywords', [])`。
5. **改代码后线上不生效**：Python 模块导入在启动时绑定，改了 router.py / skill JSON 必须重启 agent_final.py 进程。
6. **L7 约束误杀天气技能**（2026-03-20）：用户说"天气不对吧"，L7 自动生成 `block_skill: weather` 约束（patterns: `['今天*']`），把所有"今天"开头的天气查询全拦了。根因：L7 约束生成逻辑把"数据不准"误判为"技能不该调用"，应该是让技能重新查询而非禁用技能。修复：清理错误约束。
7. **规则层过度自信架空 LLM**（2026-03-20）：2 字关键词降权 + classify_stage 误判 explore + 意图门控 0.75 三者叠加，导致"今天天气怎么样""最新新闻"等合法请求被规则层直接判 chat，LLM 完全没机会参与裁决。详见第三轮改动。
8. **流式改造引入作用域 bug 导致技能模式 SSE 断连**（2026-03-20）：流式改造在 chat 分支内新增了 `import re as _re`（局部）和 `_stream_chunks = []`，但 skill 分支不走 chat 分支，后续代码引用这两个变量时触发 `UnboundLocalError`，SSE 生成器崩溃，前端收到断开的连接显示"后端连接失败"。修复：删除局部 `import re as _re`（用模块级的）；`_stream_chunks` 引用改为 `locals().get('_stream_chunks')`。同时 `EventSourceResponse` 加 `ping=2` 保持 SSE 心跳。

## 三、已完成的改动

### 第一轮：短期止血（2026-03-16 上午）

#### 2.1 TASK_WORDS 瘦身
原来 `TASK_WORDS` 包含 `做/写/讲/看/来个/想听/想看/想要` 这些日常口语词，导致随便说句话就加 skill_score。

改后拆成两组：
- **强任务词** TASK_WORDS：`查/生成/画/打开/搜索/搜/帮我/给我/再来一个/换一个/长一点/继续讲`
- **弱任务词** WEAK_TASK_WORDS：`做/写/讲/看/来个/想听/想看/想要` — 单独不加分，必须搭配技能关键词命中才 +0.5

#### 2.2 短关键词降权
2 字关键词（如"游戏""故事""天气"）从 +2.0 降到 +1.0。
3 字及以上关键词（如"贪吃蛇""写篇文章"）保持 +2.0。

效果：纯短关键词（如只说"游戏"）skill_score 只有 1.0，进不了 `>= 2.0` 的技能候选门槛。

> **⚠️ 已在第三轮（3.1）回调**：2 字中文词是标准词长（天气/新闻/故事），降权导致合法请求过不了门槛。现改为只对 1 字关键词降权。

#### 2.3 建议/讨论句式检测
新增 `_is_suggestion_or_discussion()` 函数，检测讨论句式。
命中后：chat_score +1.5，skill_score *= 0.5

#### 2.4 天气关键词收紧
weather.json 里城市名作为关键词，导致"我在常州"触发天气查询。
改后：城市名命中时，必须同时存在天气动作词（天气/气温/温度/下雨/晴天/多少度等）才算。

#### 2.5 LLM lite hint 过滤
route_resolver.py 里，只有 confidence >= 0.7 的关键词候选才传给 LLM lite 当 hint。

#### 2.6 弱技能信号分流
- 有 matched_skill → hybrid 0.45（低置信度，交 LLM 确认）
- 无 matched_skill（只有任务词没有技能关键词）→ chat 0.7

### 第二轮：意图分层 + 执行许可（2026-03-16 下午）

#### 2.7 新增 classify_intent() — core/router.py
在 `route()` 的通用技能候选层前加意图门控，把用户输入分为 4 类：

| 意图 | 含义 | 示例 | 后续处理 |
|------|------|------|----------|
| `task` | 明确执行请求 | "帮我查天气""写篇文章" | 允许进技能匹配 |
| `discuss` | 讨论/建议/假设 | "或者玩游戏都可以""如果要画海报" | 走 chat，不进技能 |
| `inform` | 陈述事实/提供信息 | "我在常州""我叫彬哥" | 走 chat，不进技能 |
| `chat` | 闲聊/问候/情感 | "你好""哈哈""累了" | 走 chat |

分类规则（纯规则，不调 LLM）：
1. 强假设词（如果/假如/万一）→ 无条件返回 discuss
2. 讨论词命中 且 不同时有(任务词+技能关键词) → discuss
3. 探询词命中 且 无任务词 且 无技能关键词 → discuss
4. 自述模式（我在X/我叫X）且无任务词 → inform
5. 组合证据 >= 2 项 → task（见下方 2.8）
6. 以上都不是 → chat

#### 2.8 组合证据门槛
`task` 意图需要同时满足至少 2 项：
- A. 有动作词（TASK_WORDS）
- B. 有请求结构（帮我/给我/请你/帮忙/一下/能不能帮/可以帮）
- C. 有明确技能关键词命中（matched_skill != None）

特殊加分：
- 弱任务词 + 技能关键词 → evidence 至少 2
- 多个技能关键词命中（skill_score >= 2.0）→ evidence 至少 2（解决"常州天气怎么样"）

#### 2.9 反例库（anti_keywords）
每个技能 JSON 新增 `anti_keywords` 字段，命中时整个技能跳过匹配。

已添加反例的技能：
- `story.json`：讲个道理、故事性、故事结构等
- `weather.json`：我在北京、天气功能、做个天气等
- `run_code.json`：游戏行业、游戏好不好玩、或者玩游戏等
- `draw.json`：海报设计、画图软件、如果要画等
- `news.json`：新闻行业、新闻学、做个新闻等
- `stock.json`：股票市场怎么样、炒股技巧、如果买股票等
- `article.json`：文章结构、文章怎么写、如果写文章等

代码改动：
- `core/skills/__init__.py`：注册时加载 `anti_keywords` 字段
- `core/router.py` `_score_text()`：遍历技能时先检查 anti_keywords，命中则 continue

#### 2.10 意图门控 — core/route_resolver.py
在 resolve_route() 阶段 1 之后、阶段 2 之前，新增意图门控：
- 如果 core_route 的 intent 是 discuss 或 inform，且 confidence >= 0.75
- 直接返回 core_route，不让 LLM lite 翻盘

解决问题："头脑风暴吧 或者玩游戏都可以" router 正确返回 chat/discuss（conf 0.85），但 0.85 < 0.9 进了 LLM lite，LLM 返回 story。加门控后直接走 chat。

> **⚠️ 已在第三轮（3.5）回调**：0.75 门控过于激进，把"天气怎么样"（被误判为 discuss）等合法请求也拦了，LLM 完全没机会参与。现改为 >= 0.9 才短路，0.75-0.9 的灰色地带交给 LLM 裁决。

#### 2.11 保守兜底 — core/route_resolver.py
阶段 3 兜底改为保守回退：
- LLM lite 失败时，只有 confidence >= 0.85 才信关键词结果
- 否则默认走 chat（不确定时宁可不执行，也不要误执行）

### 第三轮：恢复 LLM 裁决权 + 修正规则层准确性（2026-03-20）

**背景**：第二轮为了防误触发，规则层越收越紧，最终把 LLM 的判断权架空了。"今天天气怎么样""最新新闻""帮我画一张猫"等合法请求全被规则层判成 chat，LLM 完全没机会参与。核心矛盾：规则层分类不准 + 意图门控过于激进 = 双重阻断。

**修复原则**：提高规则层准确性，同时放宽门控让 LLM 裁决灰色地带。

#### 3.1 2 字关键词恢复正常评分 — core/router.py `_score_text()`
中文 2 字词是标准词长（天气/新闻/故事/股票），和英文 5-7 字母等价，不该降权。

改前：`len(kw) < 3` → +1.0（2 字关键词被降权）
改后：`len(kw) < 2` → +1.0（只对 1 字关键词降权）

效果："天气"单独命中 skill_score = 2.0，能过技能候选门槛。

#### 3.2 explore 门控不再拦截有技能关键词的输入 — core/router.py `classify_stage()` P3
"天气怎么样"里的"怎么样"命中 EXPLORE_WORDS，但用户是在问天气，不是在探索讨论。

改前：`not (has_task and has_skill_kw) and not strong_skill` → 需要同时有任务词+技能关键词才豁免
改后：`not has_skill_kw and not strong_skill` → 有技能关键词就豁免

效果："天气怎么样"不再被归为 explore/discuss。

#### 3.3 技能关键词命中 = 隐式请求 — core/router.py `classify_stage()` P4 + `classify_intent()`
"最新新闻""明天下雨吗"没有"帮我/查一下"等任务词，但用户意图很明确。

新增规则：技能关键词命中 + 无闲聊/情绪反信号（chat_score <= 0 且 emotion_score <= 0）→ evidence 至少 2，视为隐式请求。

效果："最新新闻"（只有技能关键词，无任务词）→ evidence=2 → request → 走技能。

同步修改 `classify_intent()` 的 discuss 门控：有技能关键词时不归 discuss（和 classify_stage P3 保持一致）。

#### 3.4 技能关键词补全 — core/skills/*.json
- `weather.json`：补 多少度/冷不冷/热不热/穿什么（原来只在 WEATHER_ACTION_WORDS 里，不在技能关键词里）
- `draw.json`：补 画一张/画一个/画一幅/画个/帮我画（原来只有"画图""海报"等，"帮我画一张猫"匹配不到）

#### 3.5 意图门控放宽 — core/route_resolver.py
改前：discuss/inform + conf >= 0.75 → 直接返回，不让 LLM 翻盘
改后：discuss/inform + conf >= 0.9 → 才短路

效果：0.75-0.9 的灰色地带交给 LLM 裁决，恢复 LLM 的判断权。配合 3.2 的 classify_stage 修正，规则层不再把合法请求误判为 discuss，所以 LLM 翻盘 discuss 的老问题（坑 #3）不会复发。

#### 3.6 置信度门槛同步 — core/router.py
技能候选层的 `kw_len >= 3` 改为 `kw_len >= 2`，和 3.1 的评分逻辑保持一致。

#### 3.7 L7 错误约束清理 — memory_db/feedback_rules.json
清理了 2 条错误的 `block_skill: weather` 约束：
- `patterns: ['今天*']`（用户说"天气不对吧"生成的，拦了所有"今天"开头的天气查询）
- `keywords: ['度']`（用户说"不对啊你都没查"生成的，拦了所有含"度"的天气查询）

根因：L7 约束生成逻辑把"数据不准"误判为"技能不该调用"。后续需要改进 L7 的约束生成策略。

#### 3.8 SSE 作用域 bug 修复 — routes/chat.py

流式改造在 chat 模式的 if 分支内新增了两个变量，但 skill 模式不走该分支，后续代码引用时触发 `UnboundLocalError`，SSE 生成器崩溃，前端显示"后端连接失败"：

- **`_re` 变量**（line 686）：chat 分支内 `import re as _re`，Python 将整个函数的 `_re` 视为局部变量，skill 分支未赋值 → line 757/790 引用时崩溃。修复：删除局部 import，使用模块级 `import re as _re`（line 9）。
- **`_stream_chunks` 变量**（line 718）：chat 分支内 `_stream_chunks = []`，skill 分支未赋值 → line 807 `if not _stream_chunks` 崩溃。修复：改为 `locals().get('_stream_chunks')`。

**教训**：在 async generator 的 if/else 分支内赋值变量时，要确保所有分支都能访问到，或者在分支外初始化默认值。

#### 3.9 SSE 心跳 — routes/chat.py

`EventSourceResponse(event_stream())` 改为 `EventSourceResponse(event_stream(), ping=2)`。

sse_starlette 的 `ping` 参数在独立 task 中每 N 秒发 SSE 注释（`: ping`），不受生成器阻塞影响。技能执行期间（5-8 秒无事件）保持连接活跃，防止客户端/代理判定超时断连。

#### 3.10 技能执行异步化 — routes/chat.py

`unified_skill_reply(bundle, skill, rewritten_input)` 改为 `await asyncio.to_thread(unified_skill_reply, ...)`，与 chat 模式保持一致，避免同步 LLM 调用阻塞事件循环。

## 五、未完成的优化（后续方向）

### 5.1 句子成分拆解（已部分实现）
`analyze_compound()` 已实现复合句拆解（按标点/连词分句，逐句分类）。
单句内的成分拆解（背景/参数/动作）尚未实现。

### 5.2 执行与回复分离
当前技能执行和回复生成是耦合的。
目标：技能只返回结构化数据，回复由人格层统一润色。
当前状态：部分技能（news）已做了分离，其他技能还是直接返回文本。

### 5.3 LLM 路由质量提升
llm_route_lite 已有 10 条 few-shot 示例，但仍有改进空间：

### 5.4 L7 约束生成策略改进
当前 L7 对"数据不准"类反馈会生成 `block_skill` 约束，直接禁用技能。
应该区分：
- "技能不该调用"（如误触发）→ block_skill
- "技能结果不准"（如天气数据过期）→ 不应禁用技能，应标记为"需要重新查询"

### 5.5 规则层与 LLM 层的职责边界
**核心原则**：规则层负责快筛（高置信度直通 + 明确排除），LLM 负责灰色地带裁决。
**警惕倾向**：为了堵具体 case 不断收紧规则层门控 → 最终架空 LLM → 系统变傻。
**检查标准**：每次加规则门控时，问自己"这个 case 是否可以通过改进 LLM prompt 解决？"如果可以，优先改 prompt 而非加规则。
