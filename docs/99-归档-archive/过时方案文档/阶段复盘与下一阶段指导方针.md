# NovaCore 阶段复盘与下一阶段指导方针

最后更新：2026-03-16 02:42

## 一、项目目标回顾

NovaCore 要做的是一个能"自主学习、自我修复、持续进化"的本地 AI 工作台。核心闭环：

```
用户对话 → 路由判断 → 技能执行或聊天回复 → 用户反馈
    ↑                                           ↓
  知识回流 ← L8 自主补学 ← L7 负反馈纠偏 ← 反馈记录
                ↓
        自我修复提案 → 最小验证 → 自动/人工应用
```

不只是聊天机器人，而是一个能从错误中学习、能修改自己源码的 Agent 系统。

## 三、本次完成的工作

### 3.1 前端模块化拆分（第三阶段）

核心问题：`output.html` 3003 行单文件，样式、脚本、结构混在一起，改一处容易影响别处。

改动：

1. 提取全部 CSS 到 `static/css/main.css`
2. 按职责拆分 JS 为 7 个独立模块（utils/awareness/chat/memory/settings/docs/app）
3. `output.html` 精简为 78 行纯 HTML 骨架 + script 标签引用
4. `agent_final.py` 新增 FastAPI StaticFiles 挂载
5. 跨模块共享函数统一到 `utils.js`，消除重复定义

涉及文件：`output.html`、`agent_final.py`、`static/css/main.css`、`static/js/*.js`

### 3.2 前端思考模式重构（此前完成）

核心问题：前端 `send()` 函数调用 `buildProcessTrace(text, replyText)` 做客户端关键词匹配生成思考气泡，完全忽略后端 `/chat` 返回的 `data.trace` 数据。

改动：

1. 后端 `build_trace_payload()` 增强为生成 2 张卡片（理解 + 路径），chat 模式也返回 `show: True`
2. `prettify_trace_reason()` 补了 chat fallback，空 reason 时也能给出合理文案
3. 前端 `send()` 改为直接使用 `data.trace.cards`，不再做客户端关键词匹配
4. 删除约 100 行废弃的前端 trace 生成代码（`buildProcessTrace` 及 4 个 extract 辅助函数）
5. CSS 加了 `.thinking-bubble-label:empty` 隐藏规则

涉及文件：`core/reply_formatter.py`、`output.html`、`tests/test_trace_payload.py`

## 四、当前存在的问题

### 4.1 前端已拆分但仍有改进空间

前端已从单文件拆成 8 个独立模块，维护成本大幅降低。但仍然大量使用 `innerHTML` 拼接，没有前端测试覆盖，设置页 554 行仍然偏大。

### 4.2 前后端数据契约不明确

后端 `/chat` 返回 `{reply, trace, repair}` 三个字段，但前端之前只用了 `reply`，`trace` 和 `repair` 都被丢弃。后端加了字段前端不知道要用，缺少一份明确的接口契约文档。

### 4.3 学习系统精准度不足

L8 自主学习依赖 Bing RSS 搜索，搜索结果有时不够精准（如搜 MCP 协议会混入 Minecraft 内容）。知识库 30+ 条记录质量参差不齐，没有自动清洗或质量评估机制。

### 4.4 自动生成技能仍然关闭

`allow_skill_generation: false`——进化闭环里最关键的一环，因风险太高一直没打开。系统的"进化"目前只停留在知识层面，还不能真正长出新能力。

### 4.5 测试覆盖有盲区

51 个测试主要覆盖路由、技能、反馈分类、自我修复等后端逻辑。前端零测试，设置页复杂交互链路没有回归保障。`agent_final.py` 的 18 个 HTTP 端点也没有集成测试。

### 4.6 部署和环境管理缺失

没有 `requirements.txt` 或 `pyproject.toml`，`desktop.py` 硬编码了本机 Python 路径。只能在当前机器上跑，换一台就要手动调。

## 五、下一阶段指导方针

### 原则一：前端展示层是当前最值得投入的方向

后端的 trace、repair、学习状态这些数据已经做好了，但用户在界面上感知不到。优先让后端已有能力在前端被看到、被感知到。

### 原则二：建立前后端接口契约

为 `/chat`、`/docs/index`、`/self_repair/*` 等核心接口补一份明确的数据结构说明。后端每加一个字段，前端必须有对应的消费逻辑，不能再出现"后端返回了但前端丢弃"的情况。

### 原则三：前端拆分已完成，后续关注质量提升

`output.html` 已拆成 8 个独立模块。下一步关注：
- 设置页 `settings.js`（554 行）可进一步按区块拆分
- 逐步减少 `innerHTML` 拼接，考虑模板化方案
- 为关键交互补前端回归测试

### 原则四：补关键路径的测试

优先补：
- `/chat` 端点的集成测试（验证 reply + trace + repair 三个字段都正确返回）
- 设置页关键交互的回归测试（学习强度切换、修复提案操作）
- 前端至少补 trace 渲染的基本断言

### 原则五：学习系统质量提升

- 给知识库条目加质量评分或过期机制
- 搜索结果增加相关性过滤
- 为 `allow_skill_generation` 设计安全的渐进开放方案（如：只允许生成特定类型的技能、生成后必须人工审批）

### 原则六：工程化基础设施

- 补 `requirements.txt` 或 `pyproject.toml`
- `desktop.py` 改为自动检测 Python 路径
- 考虑加一个最小的 CI 流程（至少跑测试）

## 六、当前架构快照

```
desktop.py → agent_final.py (FastAPI, port 8090) → output.html
                                                     ├── static/css/main.css
                                                     └── static/js/ (utils/awareness/chat/memory/settings/docs/app)

agent_final.py 导入链：
  core/json_store.py        → 公共 JSON 读写
  core/state_loader.py      → 状态加载、路径常量、文档索引
  core/context_builder.py   → 对话上下文组装（L1-L8）
  core/route_resolver.py    → 多阶段路由决策
  core/reply_formatter.py   → 回复格式化、trace 构建
  core/feedback_loop.py     → 反馈记录、后台学习任务

其他核心模块：
  core/router.py            → 规则优先路由
  core/executor.py          → 技能执行分发
  core/l8_learn.py          → L8 自主学习
  core/self_repair.py       → 自我修复提案
  core/feedback_classifier.py → 反馈分类
  brain/                    → 人格风格、LLM 调用
  core/skills/              → 天气、故事、代码、绘图、股票
  memory_db/                → 统一持久化目录
```
