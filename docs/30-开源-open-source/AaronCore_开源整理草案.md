# AaronCore 开源整理草案

最后更新：2026-04-14

这份文档只回答三件事：

1. AaronCore 对外应该怎么讲
2. 哪些内容应该进入公开仓库
3. 哪些内容必须继续留在私有开发仓

## 一句话定位

AaronCore 不是“把记忆文件读回来”的 agent，而是一个把记忆、人物延续、用户关系和任务状态接进主运行链的 memory-first agent project。

更稳妥的公开表达：

- memory-first agent project
- single-window continuity
- memory participates in the current turn, not only in storage
- task state is being connected into the same runtime substrate

不建议上来就说：

- 全面超过某某项目
- 最强 agent
- 已经解决通用任务连续性的一切问题

## 公开仓与私有仓的职责

### 私有开发仓

继续承担真实开发和真实使用：

- 真实记忆
- 真实状态
- 真实任务历史
- 真实实验节奏
- 本地秘钥与个人配置
- 尚未整理的中间实现

这个仓库可以继续“脏”，因为它首先服务于开发者本人和真实迭代。

### 公开仓

只承担对外协作和公开叙事：

- 干净、可运行的代码
- 公开文档
- 模板化配置
- 空模板或示例数据
- 安装、运行、贡献说明

公开仓不应该承载开发者本人的真实运行痕迹。

## 什么该同步，什么不该同步

核心原则：

**模板同步，真值不同步。**

### 应该同步到公开仓的内容

- 源代码
- 架构文档
- 默认配置
- 配置模板
- 空的状态模板
- 演示样例
- 安装脚本
- 测试

### 不应该同步到公开仓的内容

- 真实记忆内容
- 真实任务状态
- 真实对话快照
- 个人偏好真值
- 用户画像真值
- 自动学习产出的私有沉淀
- 本地模型秘钥
- 搜索秘钥
- 调试备份
- 仅对当前开发者有意义的运行产物

## 当前仓库里的公开化边界

现阶段最适合的策略不是“把当前私库直接公开”，而是：

1. 私库继续作为主开发仓
2. 新建一个干净的公开仓
3. 从私库向公开仓做阶段性整理同步

同步单位应该是“可解释的小版本”，而不是每天的全部真实快照。

## 配置与状态的拆分原则

公开仓应该尽量采用：

- `public defaults`
- `local override`
- `ignored truth data`

适合公开的，是结构、字段、模板、默认值。

不适合公开的，是个人化真值、真实历史、秘钥和运行中间态。

## AaronCore 开源时最该强调的差异

与常见 agent 项目相比，AaronCore 更适合强调以下几件事：

1. 记忆不是外挂模块，而是主链的一部分
2. 连续性不是“重新读档”，而是当前轮仍然活在同一条关系与任务链上
3. 任务状态正在与记忆层接通，而不是完全割裂

更适合展示的三个场景：

1. 长时间单窗口使用，不开新对话
2. 被打断后继续任务，不需要重新交代背景
3. 对人的偏好、关系和项目状态自然延续

## 协作边界

AaronCore 可以开源，但不应该让所有人都随意改主链。

建议边界：

- 外围能力开放协作
- 核心主链谨慎合并
- 涉及 memory / continuity / runtime 的改动必须高标准审查
- 主链方向和最终合并权保留在项目维护者手里

## 公开仓的第一版目标

第一版公开仓不追求“把所有能力都放出去”，而追求三点：

1. 别人能跑起来
2. 别人能看懂 AaronCore 的方向
3. 别人能在不接触私有真值的前提下参与贡献

## 下一步整理清单

- 补一个根目录 `README.md`
- 准备公开仓的 `LICENSE`
- 准备 `CONTRIBUTING.md`
- 梳理公开仓需要保留的 `state_data` 模板
- 继续把秘钥与本地真值从可追踪文件中拆出去
- 明确第一版公开仓的最小可运行范围

## 当前建议

现在不需要立刻把私库公开。

当前更合理的动作是：

- 在私库里继续开发
- 同时整理公开版骨架
- 等公开边界稳定后，再创建真正的 AaronCore 公库

这样既不破坏当前开发体验，也不会错过以后面向世界公开的机会。
