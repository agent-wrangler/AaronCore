NovaCore / 当前实际运行流

Agent 项目当前流程图

这份图是按你现在工作区里的代码梳出来的当前主链，不是旧文档里的理想流程。 目前核心是：前端把消息发到 /chat，后端先组上下文，再交给 LLM 做原生 tool_call 决策。首轮就会暴露真实技能工具，discover_tools 只是兜底； 技能层下面真正更强的底座，是 6 层协议能力。

启动层 路由与上下文层 工具执行层 记忆闭环层

一眼看懂主链

启动层

1. 桌面壳启动

从 start_nova.bat 进入 Electron 壳。壳会先检查本地 8090 端口，如果后端没起，就自动拉起 agent_final.py。

  • start_nova.bat
  • shell/main.js
  • agent_final.py
路由层

2. 前端发起对话

前端页面通过 fetch('/chat') 建立 SSE 流式连接，边收 trace，边收 stream token，最后收完整 reply。

  • output.html
  • static/js/chat.js
  • SSE 事件：trace / stream / reply / repair / ask_user
上下文层

3. 后端组装 bundle

routes/chat.py 收到消息后，先写历史，再装当前轮上下文。当前主链默认是 tool_call + CoD。

  • L1 最近对话
  • L2 session_context
  • L4 persona
  • dialogue_context
  • flashback_hint
  • L7 relevant rules
工具层

4. LLM 首轮就看到真实工具面

系统不是先规则分流，而是直接把工具列表交给模型。模型自行判断是直接回答，还是发起一个或多个 tool_call。

  • 首轮直接暴露真实技能工具
  • 记忆工具：recall_memory
  • 知识工具：query_knowledge
  • 诊断工具：read_file / self_fix
  • 交互工具：ask_user
  • discover_tools：只做兜底，不是当前主入口
执行层

5. 技能层再落到协议底座

工具通过 core/tool_adapter.py 桥接到执行器。技能类工具走 core/executor.py，而电脑操作类技能继续落到 6 层协议底座。记忆/搜索/自修复工具则走内部专用路径。

  • 技能工具 -> execute()
  • 协议技能 -> target_protocol / fs_protocol / app / window / UI / repair
  • 内部工具 -> 独立 handler
  • 结果回喂给 LLM
多轮层

6. 多轮 tool_call 循环

不是单次调用。模型在拿到上一步工具结果后，还可以继续调用下一轮工具，直到它决定停下并输出最终文本。

  • 主入口：unified_reply_with_tools_stream()
  • 安全上限：20 轮
  • 支持边执行边把状态推回前端
回写层

7. 回复后写回记忆

一轮回复结束后，会进入记忆和反馈闭环，不只是把文本发出去就结束。

  • L7 记录负反馈和纠偏规则
  • L8 可触发自动补学
  • L2 把本轮对话加入持久记忆
结论层

8. 当前真实定位

你这版项目的本质更像一个“LLM 主导的桌面 Agent”。技能层不是最终底座，它更像协议底座对上暴露出来的一层能力接口。

  • 路由权交给 LLM
  • CoD 只预装轻量上下文
  • 首轮就能看到真实技能工具
  • 6 层协议底座才是电脑操作核心能力层
  • 记忆和知识按需拉取

单次消息细流程

一次完整请求是怎么走完的

1
前端发送消息

static/js/chat.js 发起 fetch('/chat')，同时准备接收 SSE 流式事件。

2
后端先写历史，再装 bundle

routes/chat.py 会先把用户消息记到历史里，然后组本轮上下文：L1、L2 session_context、L4、dialogue_context、flashback、L7。

3
构建 tools 列表

当前默认走 CoD。首轮就会暴露真实技能工具，加上 recall、knowledge、文件读写、自修复、ask_user 等内部工具；discover_tools 只是兜底。

4
LLM 第一次决策

unified_reply_with_tools_stream() 发起第一轮 LLM 调用。模型要么直接回文本，要么返回一个 tool_call。

5
工具执行并回喂

core/tool_adapter.py 执行工具。若是电脑操作类技能，会继续落到 executor 和 6 层协议底座，再把结果拼回 messages 交给模型继续决策。

6
多轮循环直到停下

模型最多可连续调用 20 轮工具。只要它还在发 tool_call，就继续执行；停下时输出最终回复。

7
前端呈现 trace / stream / reply

前端边显示步骤状态，边渲染流式文本，最后用 reply 事件做一次最终定稿。

8
回写闭环

回复结束后进入 L7 反馈记录、L8 自动补学、L2 add_memory，这也是 NovaCore 和普通聊天页最大的差异之一。

当前默认预装 vs 按需拉取

默认预装 L1 / L2 session / L4 / dialogue_context / L7

这些内容会直接进入本轮 bundle，用来帮助模型立刻理解当前对话、语气和承接关系。

按需拉取 L2 持久记忆 / L3 / L5 / L8

这些不再全量塞进 prompt，而是通过 recall_memory、query_knowledge 这类工具按需检索。

额外联想层 flashback_hint

这层不是历史摘要，而是“如果当前情绪或话题触发了旧记忆联想，就给模型一个可忽略的提示”。

真实能力底座 技能层之下还有 6 层协议层

技能层更像对 LLM 暴露出来的接口壳。真正负责目标解析、状态校验、应用/窗口/UI 操作与修复闭环的，是下面那层协议底座。

现役链路 vs 历史残留

现在真正主导运行的部分

现役 routes/chat.py 的 tool_call 主链

现在主路由就是它。真正的决策权在 LLM，不再是旧式规则路由先挑技能。

现役 core/reply_formatter.py 的多轮 tool_call

这里负责和模型交互、处理 tool_call、把结果流式回推，是当前最关键的执行枢纽之一。

现役 tool_adapter + executor + 6 层协议底座

当前真正的能力闭环不是只到 skills 为止，而是一路落到 target_protocol / fs_protocol / app / window / UI / repair 这些协议层。

目录里还在，但别误当主链的部分

历史残留 非 tool_call 路径

routes/chat.py 里仍保留了旧的“协议前置直达派发挂法”，但当前默认不会走到。旧的是这条挂法，不是 6 层协议底座本身。

历史残留 archive / docs/99-归档-archive

这些内容有参考价值，但不能当作当前真实运行态。讨论“现在系统怎么跑”时应以当前代码为准。

历史残留 旧式规则分流心智模型

这是最容易让人两眼一黑的点。当前项目已经不适合再用“先路由、再技能、最后 LLM 兜底”的思路理解。

6 层协议底座

第 1 层

环境感知层

负责“我现在面对的到底是什么目标和环境”。这是协议底座的入口层，不直接替 LLM 做路由，而是提供可操作的现状判断。

  • 当前窗口和窗口列表感知
  • 目标类型识别：url / path / app / window
  • 导出状态续接与上下文续接
  • 核心文件：core/target_protocol.py
第 2 层

文件系统协议层

负责文件、目录、保存目标、危险后缀、后置校验和文件类漂移修复，是“文件到底落没落盘”的真实判定层。

  • open/save/delete/copy/move 后置校验
  • 危险目标识别与确认
  • 保存路径、桌面回退、导出状态记录
  • 核心文件：core/fs_protocol.py
第 3 层

应用生命周期协议层

负责应用启动、关闭、窗口出现验证、聚焦恢复。技能层里的 app_target 其实是这一层对上暴露的入口。

  • 启动应用、结束应用、检测新窗口
  • 聚焦已有窗口而不是重复拉起
  • 窗口未出现时的重试与失败判定
  • 核心文件：tools/agent/app_target.py
第 4 层

窗口管理协议层

负责窗口状态变更本身，比如最小化、最大化、还原、移动、缩放、焦点切换。这层关注的是窗口，不是控件内容。

  • 窗口焦点、移动、缩放、最小化、最大化
  • 窗口标题匹配和目标窗口定位
  • 依赖目标解析层提供窗口候选
  • 核心文件：tools/agent/ui_interaction.py
第 5 层

UI 交互协议层

负责点击、输入、提交、快捷键和控件级交互。浏览器场景会优先尝试更精确的 CDP / Accessibility 路径，再回退到通用操作。

  • 点击、输入、点击并提交
  • 浏览器 CDP 交互、桌面控件 Accessibility 交互
  • 必要时回退到通用 UI 操作
  • 核心文件：tools/agent/ui_interaction.py
第 6 层

执行编排层

负责把“执行 -> 校验 -> drift -> repair -> 再执行”串成闭环。真正让这套协议底座扎实的，不是能做，而是做完能验、出偏差能修。

  • 技能统一执行分发
  • post-condition 校验
  • drift 检测与自动修复
  • 核心文件：core/executor.py、core/fs_protocol.py

8 层认知与记忆支撑结构

L1

原始对话窗口

这是当前主链里最直接的短期上下文来源，保存原始 user / assistant 消息，不是摘要层。

  • 最近原始消息窗口
  • 给 tool_call 主链喂原始 messages
  • 核心文件：memory_db/msg_history.json
L2

会话态 + 持久记忆中枢

当前要拆开理解：默认预装的是轻量会话态，持久记忆内容在 CoD 下主要按需拉取。

  • session_context：当前会话态
  • l2_memories：按需回忆的持久记忆
  • 核心文件：core/l2_memory.py、core/session_context.py
L3

经历记忆

负责“发生过什么”，记录共同经历、事件、里程碑。当前 CoD 下默认不全量预装。

  • event / milestone / general
  • 按需回忆为主
  • 核心存储：memory_db/long_term.json
L4

人格与用户画像

这是当前最稳定、最强预装的一层。它不负责知识检索，负责的是“Nova 是谁、用户是谁、怎么相处”。

  • 人格设定、说话风格、交互规则
  • 用户画像与关系信息
  • 核心存储：persona.json / L4 读取逻辑
L5

成功执行经验 / 工作方法记忆

它已经不再承担“技能列表”职责。当前更有价值的是沉淀经过验证的成功执行路径，在复杂任务里按需给规划层提供可复用的方法参考。

  • 来自 L6 提炼后的稳定成功经验
  • 默认不全量进入 prompt，只在复杂任务里按需召回
  • 更像“这类事怎么做更稳”，而不是“我会哪些技能”
L6

原始执行轨迹 / 素材层

它不是普通记忆，也不是 prompt 主体，而是真实执行后的素材层。这里留下 success、verified、drift、observed_state 等事实，供后续提炼成功经验与失败纠偏。

  • 真实 tool run 的执行事实
  • L5 成功经验与后续 L7 反思的源数据
  • 主要在回复后更新，不直接主导当前 prompt
L7

反馈纠偏层

负责记录“以前哪里答错了、以后要怎么避免重犯”。当前主链会预装少量 relevant rules。

  • 负反馈、纠偏规则、经验教训
  • 回复后可能触发再学习
  • 核心文件：core/feedback_loop.py
L8

已学知识层

负责系统学到的知识卡片。当前 CoD 下默认按需查询，不再一上来把整层塞进 prompt。

  • 知识检索与知识沉淀
  • 命中时优先复用已学知识
  • 核心文件：core/l8_learn.py

当前最值得盯住的风险点

风险 1

代码和文档容易错位

项目迭代很快，文档和历史方案非常多。如果每次不先确认“当前真实在跑哪条链”，分析很容易串版本。

  • 优先看当前入口和主函数
  • 归档文档只当参考，不当事实
风险 2

协议底座容易被技能层遮住

现在最容易发生的误判，不是“看不见技能”，而是只看见技能，没看见技能下面还有更强的协议底座。

  • 技能是接口层，不是最终能力层
  • 协议层负责解析、校验、修复和闭环
风险 3

上下文层边界容易混

L1、dialogue_context、flashback 看起来都像“上下文”，但职责不同，改错一层就容易出现重复、跑偏或说两遍。

  • L1 = 原始对话窗口
  • dialogue_context = 增量提示
  • flashback = 旧记忆联想

最新底层修正

动作结果协议

6 层底座内部新增统一动作语义

这轮没有新增第 7 层，而是在现有 6 层协议底座内部补强了“动作结果协议 / 执行闭环协议”。网页、应用、写文件这类通用动作开始统一返回结构化结果，而不是只靠技能自己吐一句话。

  • action_kind / target_kind / outcome / display_hint
  • 步骤层开始优先读取协议层动作摘要
  • 核心文件：core/fs_protocol.py、core/executor.py
  • verification_mode / verification_detail 也开始纳入统一动作结果
  • verification_detail 现在已经顺着 run_event -> L6 -> 前端记忆页 往下透传
  • write_file 现在更应该被理解成“底座协议动作的对外接口”，不是零散业务技能
  • 凡是通用型能力，都应该尽量下沉到底座，变成所有技能可复用的共用能力
MiniMax 修正

空回复根因在 tool-result 回填链

最近那种“工具已经成功，但最后 MiniMax 空内容”的问题，根因不是模型不肯答，而是 assistant + tool_calls 消息在标准化层里被吞掉，导致第二轮回填时 tool id 对不上。

  • 关键修正点：brain/__init__.py
  • 错误形态：tool result's tool id not found (2013)
  • 经验：先查协议链，再考虑兜底
收尾修正

动作已完成时，最后一跳必须稳定收尾

最近又暴露出另一类问题：工具其实已经成功，但模型最后只剩一句“我来处理 / 马上就好”这类前导话，或者把旧式 <minimax:tool_call> / <invoke ...> 标记混进最终回复，前端体感就会像卡住没回话。

  • 收口位置：core/reply_formatter.py
  • 原则：不改 routes/chat.py 主顺序，只修成功后的收尾桥接
  • 策略：先清掉旧式内部 tool 标记，再判断是否需要用 action_summary / verification_detail / tool_response 自动补收尾
  • 目标：动作做完时，用户一定能看到一条干净、可见、可交付的结果回复
单实例锁

客户端壳层不再允许重复开窗

Electron 壳现在补了单实例锁。重复启动 Nova 时，不再新开第二只客户端，而是恢复并聚焦现有窗口。这属于入口层基础约束，不是多 Agent 能力。

  • 核心文件：shell/main.js
  • 第一次启动：正常创建窗口
  • 第二次启动：直接唤醒现有窗口
